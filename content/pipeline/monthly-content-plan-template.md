## 4. Поля согласования и стоимости

| Поле | Как заполнять |
|---|---|
| `Approval status` | `draft`, `review`, `needs_changes`, `approved`, `rejected` |
| `Generation cost rub` | `0` для dry-run, дальше фактическая стоимость генерации |
| `Model tier` | `dry_run`, `cheap`, `strong` |
| `Regeneration reason` | `tone`, `facts`, `duplicate`, `CTA`, `legal_risk`, `visual`, `other` |
| `Regeneration notes` | Простыми словами: что именно нужно переделать |

## 5. Статусы

Используем только эти статусы:

```text
idea -> planned -> draft -> review -> needs_changes -> approved -> scheduled -> published -> analyzed
```

Правило:

- `approved` не значит “автоматически публиковать”.
- Реальная публикация только после отдельного подтверждения.
- PostMyPost и автопубликация пока отложены.

## 6. Канальные правила

| Канал | Роль канала | Что важно |
|---|---|---|
| MAX | B2C доверие и мягкий вход | Простой язык, бытовые риски, без перегруза |
| Telegram | B2B экспертность | Практика ГИПа, инженерные детали, партнёрство |
| VK | Социальное доказательство | Кейсы, команда, процесс, живой тон |
| Дзен | SEO и длинные статьи | Инструкции, структура, поисковые формулировки |
| Яндекс Карты / 2GIS | Локальное доверие | Новости, фото, отзывы, аккуратные обновления |

## 7. Проверка перед публикацией

Перед статусом `approved` проверить:

- нет персональных данных;
- нет обязательной формы захвата контакта;
- нет одинакового текста во все каналы;
- есть понятный следующий шаг;
- текст не звучит как шаблонный AI;
- тема связана с гипотезой месяца.
