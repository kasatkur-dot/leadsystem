# Воронка выпуска постов для lead-engine

Дата: 2026-05-16

Статус: MVP-регламент. Реальная автопубликация не включена.

## 1. Зачем нужна воронка

Посты не должны выходить просто потому, что появилась хорошая тема.

Каждый пост должен вести человека на один маленький следующий шаг:

```text
узнал свою проблему
-> понял риск
-> увидел порядок работы
-> доверил первичную проверку
-> прислал план / вопрос
-> попал в CRM как заявка или теплый диалог
```

Цель контента в этом проекте:

- привлечь подписчиков;
- объяснить услуги проектной организации простым языком;
- получить входящие вопросы;
- получить планы квартир, описание объекта или запрос на первичную оценку;
- связать пост с лидом, сделкой, выручкой и ROMI через Agent 5.

## 2. Что взято как метод

Из проекта `max-ai-automation-channel` берем механизмы, а не темы:

- пост принадлежит конкретной цепочке, а не живет отдельно;
- у каждого поста есть стадия воронки;
- CTA не должен быть сильнее готового оффера;
- нужен лимит касаний по цепочке, чтобы не греть одну тему бесконечно;
- каждую публикацию нужно записывать в журнал и потом смотреть результат;
- без `approved` нет публикации;
- автопубликацию подключать только после ручного месяца.

Из открытых источников берем только общую логику:

- 3-5 контентных опор держат стратегию сфокусированной;
- календарь отвечает на вопрос `когда`, а контентные опоры и воронка отвечают на вопрос `зачем`;
- посты должны закрывать разные стадии пути клиента: awareness, consideration, conversion, retention;
- для сложных услуг нужны доверие, доказательства и ясный следующий шаг, а не только прямой призыв купить.

Источники для методологии:

- Sprout Social: `https://sproutsocial.com/insights/insights-social-media-content-pillars/`
- Brandwatch: `https://www.brandwatch.com/social-media-glossary/content-pillars/`
- HubSpot: `https://blog.hubspot.com/marketing/content-for-every-funnel-stage`
- Semrush: `https://www.semrush.com/blog/content-marketing-funnel/`

## 3. Стадии нашей воронки

| Стадия | Простыми словами | Задача поста | Разрешенный CTA |
|---|---|---|---|
| `awareness` | человек узнает свою ситуацию | показать боль или риск | сохранить, задать вопрос |
| `diagnosis` | человек начинает проверять себя | дать признаки и чеклист | прислать план / описать ситуацию |
| `education` | человек понимает порядок | объяснить, почему нужен проект и проверка | получить первичную оценку |
| `trust` | человек оценивает студию | показать процесс, компетенцию, границы ответственности | написать в личные сообщения |
| `proof` | человек видит пример | кейс, путь клиента, безопасный разбор | прислать похожий случай |
| `conversion` | человек готов к действию | один понятный следующий шаг | ПРОВЕРКА / первичный анализ / заявка |
| `follow_up` | человек уже видел несколько постов | напомнить путь и снять страх | короткий вопрос или план объекта |

## 4. Обязательные поля каждого поста

Перед публикацией у каждого поста должны быть поля:

```text
Content ID:
Channel:
Segment:
Service line:
Funnel arc:
Funnel stage:
Offer ID:
Offer readiness:
Monetization role:
CTA intensity:
Main CTA:
Next bridge:
Metric to watch:
Status:
```

Если этих полей нет, пост можно написать как черновик, но нельзя считать его системной публикацией.

## 5. Уровни CTA

| Уровень | Что значит | Пример для ВПП |
|---:|---|---|
| 0 | без CTA | чистая польза или новость |
| 1 | мягкий вопрос | если ситуация похожа, напишите вопрос |
| 2 | самопроверка | проверьте свой план по этим пунктам |
| 3 | первичная ручная оценка | пришлите план и коротко опишите задачу |
| 4 | заявка на проект | напишите ПРОВЕРКА / нужен проект / нужна оценка |
| 5 | коммерческое предложение | только после диалога и исходных данных |

Правило: в одном посте один главный CTA.

## 5.1. Форматы и монетизация

Посты могут быть разными по форме, но каждый должен вести к измеримому результату.

Основная рабочая гипотеза для CTR:

```text
разбор живой фразы клиента
```

Пример:

```text
Клиент говорит: нам просто немного переделать здание.
Что в этой фразе опасно?
```

Дополнительные форматы:

- вопрос-ответ;
- анти-чеклист;
- дневник проектировщика;
- мини-кейс без персональных данных;
- один термин простым языком;
- два пути;
- красный флаг;
- пост-карусель / список кадров;
- позиция эксперта.

Монетизация поста может быть не только прямой продажей услуги.

Засчитываем полезный результат:

- входящий вопрос;
- присланный план или описание объекта;
- заявка в CRM;
- запрос КП после исходных данных;
- сохранение;
- репост;
- партнёрская рекомендация;
- сигнал спроса для будущего лендинга, статьи или оффера.

Запрещено считать пост успешным только потому, что он красиво написан. Нужен следующий шаг или измеримый сигнал.

## 6. Готовые офферы MVP

Главная карта: `content/pipeline/offer-readiness-map-mvp.csv`.

Сейчас можно публично предлагать:

- первичную ручную проверку плана;
- короткий ответ на вопрос по перепланировке;
- первичную оценку состава работ по проекту;
- звонок или сообщение после того, как человек прислал исходные данные.

Пока нельзя обещать:

- автоматического MAX-бота как готовый публичный продукт;
- точную цену без исходных данных;
- гарантированное согласование;
- бесплатный полный проект;
- автоматическое согласование или обход требований.

## 7. Активные цепочки

Главная карта: `content/pipeline/post-funnel-arcs-mvp.csv`.

Для ближайших публикаций активна цепочка:

```text
replan-safe-start-001
```

Ее цель:

```text
получить 3-5 входящих обращений с планом квартиры, вопросом по перепланировке или запросом на первичную оценку
```

Дополнительная цепочка:

```text
project-before-repair-001
```

Ее цель:

```text
объяснить, почему проект нужен до ремонта, стройки, сделки или согласования
```

## 8. Недельный микс

Для ручного MVP достаточно 3-4 публикаций в неделю.

| Тип | Доля | Зачем |
|---|---:|---|
| Посты активной цепочки | 60-70% | вести к заявке, а не просто информировать |
| Полезные мосты | 20-30% | новости, нормы, короткие чеклисты |
| Доверие и процесс | 10-20% | показывать работу студии и людей |

Не ставить подряд 3 поста с одинаковым CTA.

## 9. Как выбирать следующий пост

1. Открыть `content/pipeline/published-log.md`.
2. Открыть `content/pipeline/publication-tracker-mvp.csv`.
3. Проверить, какие темы уже опубликованы и что запланировано.
4. Открыть `content/pipeline/content-plan-10-posts-lead-mvp-2026-05.md`.
5. Взять ближайший пост со статусом `planned`.
6. Проверить его CTA по `offer-readiness-map-mvp.csv`.
7. Проверить роль слота по `content-slot-matrix-mvp.md`.
8. Написать черновик.
9. Отдать на ручное approval Янике.
10. После публикации записать в `published-log.md` и `publication-tracker-mvp.csv`.
11. Если пришел лид, Agent 5 связывает `content_id -> channel -> lead -> deal -> revenue`.

## 10. Метрики

Минимально считаем:

| Метрика | Где фиксировать |
|---|---|
| `views` | вручную по каналу |
| `reactions` | вручную по каналу |
| `comments_or_questions` | `published-log.md` или ретроспектива |
| `plan_sent_count` | Agent 5 / CRM / ручной лог |
| `lead_count` | Bitrix24 / Agent 5 |
| `qualified_count` | Agent 3 / Agent 5 |
| `deal_count` | Bitrix24 |
| `revenue_rub` | Agent 5 analytics |

Рабочие файлы учета:

```text
content/pipeline/publication-tracker-mvp.csv
content/pipeline/topic-response-tracker-mvp.csv
```

Пока нет фактических цифр, ставим решение:

```text
data_missing
```

Запрещено делать вывод, что тема сработала или не сработала, если нет данных.

## 11. Стоп-правила

Пост нельзя публиковать, если:

- он повторяет недавно опубликованную тему;
- нет понятного следующего шага;
- CTA ведет в неготовый оффер;
- есть обещание гарантированного результата;
- есть персональные данные, адреса, договоры, штампы, кадастровые номера;
- текст звучит как агрессивная продажа;
- пост не связан ни с одной цепочкой или полезным мостом.

## 12. Ближайший маленький шаг

Использовать план `content-plan-10-posts-lead-mvp-2026-05.md` и подготовить первый пост из цепочки `replan-safe-start-001`, но не публиковать без ручного подтверждения.

## 13. Будущие возможности

Все идеи вида `это применим позже` записывать в:

```text
content/pipeline/future-opportunities-backlog.md
```

Перед возвратом к отложенному инструменту проверять:

- что уже готово в проекте;
- какой триггер применения наступил;
- не дублирует ли инструмент Agent 4, Agent 5 или Bitrix24;
- есть ли риск секретов, платных API, автопубликации или персональных данных.

Если триггер не наступил, статус остается `later` или `watch`.
