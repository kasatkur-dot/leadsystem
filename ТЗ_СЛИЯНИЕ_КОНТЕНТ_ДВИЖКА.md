   - Все агенты работают на одном Redis и одной PostgreSQL.

## 1. Финальное состояние (определение готовности)

После выполнения этого ТЗ должно выполняться:

1. В корне `design-studio-lead-engine` есть рабочий пайплайн: сначала `.venv/bin/python -m agents.agent4_publisher.cli generate post --topic "<тема>"`, затем `.venv/bin/python -m agents.agent4_publisher.cli publish telegram --file "<готовый md-файл>"`.
2. Instagram через Postmypost пока имеет безопасный dry-run адаптер; боевую отправку включать отдельно после проверки аккаунта и токенов.
3. Генерация изображения (DALL-E 3) использует архитектурный стиль (визуализация дома, плана, разреза), а не киберпанк / Jarvis.
4. После успешной публикации Agent 4 публикует событие `content_published` в Redis-канал, и Agent 5 (CRM) его получает (минимум — пишет в лог).
5. Папка `projects/sales_ai` удалена или переименована в `_archive_sales_ai_2026-05`.
6. В `requirements.txt` lead-engine добавлены зависимости из `sales_ai` без конфликтов.
7. Все агенты Codex / Claude Code, открывая `design-studio-lead-engine/REPORT.md`, видят актуальную картину и не ходят в `sales_ai` за чем-либо.
8. Никаких **анкет / форм захвата лидов / FSM-квизов** в системе нет — захват лидов работает только через voluntary inquiry (входящие в DM, комментарии, чат).

## 5. Юридическое ограничение — без форм

**Заполнение анкет и форм захвата лидов запрещено законодательством.** Это жёсткое ограничение для всей системы.

### Запрещено

- ❌ FSM-квизы в Telegram-боте / MAX-боте с обязательными вопросами
- ❌ Лендинг с формой «оставьте email, чтобы получить PDF»
- ❌ Лид-магнит в обмен на форму
- ❌ Email-цепочка после отправки формы
- ❌ Любое поле «обязательное для заполнения» до того, как человек получит контент или ответ

### Разрешено

- ✅ Voluntary inquiry — лид сам пишет в Direct, в чат, в комментариях, на почту, звонит
- ✅ Открытый лид-магнит (PDF, чек-лист) без формы — скачивается напрямую
- ✅ Кнопка «Связаться с менеджером» — открывает диалог, без анкеты
- ✅ MAX-бот в режиме диалогового ассистента (отвечает, рассказывает, помогает) — не задаёт обязательных вопросов
- ✅ Скрытый трекинг (UTM, источник, канал, тайминг) — фиксируется автоматически на стороне CRM, без участия человека

### Что проверить в существующем коде lead-engine

- `agents/agent1_scout/` — есть ли валидаторы вида «если нет email, лид невалиден» → сделать поля опциональными
- `agents/agent3_processor/` — то же
- `agents/agent5_crm/bitrix/` — отправка в Bitrix24 должна работать, если у лида только TG-username и текст сообщения

## 8. Этапы работы

### Этап 0 — Подготовка и резервная копия (1 сессия, ~30 мин)

1. Прочитать текущий `REPORT.md` lead-engine.
2. Сделать `git commit` всех изменений в lead-engine, чтобы был чистый старт.
3. Проверить, что `.env` lead-engine заполнен (12 ключей из старого аудита).
4. Создать ветку `feature/merge-content-engine` (опционально).
5. Сделать резервную копию `sales_ai/` → `_archive_sales_ai_2026-05/` рядом или в `~/Documents/архив/`.

**Критерий приёмки:** есть резервная копия, ветка создана, `.env` готов.

### Этап 1 — Структура папок и конфиг (1 сессия, ~1 час)

1. Создать в `agents/agent4_publisher/` структуру: `core/`, `posters/`, `prompts/`, `content/`, `assets/fonts/`.
2. Создать пустые `__init__.py` во всех новых папках.
3. Создать `agents/agent4_publisher/cli.py` со скелетом argparse.
4. Создать `content/studio_brand/` и скопировать туда 3 студийных `.md`.
5. Обновить `requirements.txt` (добавить 4 пакета).
6. Обновить `.env.example` (добавить 7 переменных).
7. Скопировать шрифты в `agents/agent4_publisher/assets/fonts/`.
8. Создать README в `agents/agent4_publisher/` с описанием ролей подмодулей.
9. `pip install -r requirements.txt` без ошибок.

**Критерий приёмки:** структура папок есть, зависимости устанавливаются, импорты работают.

### Этап 2 — Перенос Python-кода (2 сессии, ~3-4 часа)

Порядок переноса — от простого к сложному:

1. `posters/telegram.py` — выделить из `sales_ai/pipeline.py` функции отправки в Telegram. Тест: отправить «hello» в тестовый канал.
2. `core/llm.py` — выделить из `sales_ai/pipeline.py` вызовы OpenRouter / Claude. Тест: получить ответ от модели.
3. `core/image.py` — перенести логику DALL-E из `generate_jarvis_dalle.py`. **Промпты не используем** — на этом этапе только проверка вызова API. Тест: сгенерировать тестовое изображение.
4. `core/voice.py` — выделить ElevenLabs из `pipeline.py`. Тест: сгенерировать голосовое.
5. `core/video.py` — выделить Replicate. Тест: сгенерировать тестовое видео (опционально, если хватает бюджета на Replicate).
6. `core/carousel.py` — перенести `generate_carousel_pil.py`. Шрифты по относительному пути. Тест: создать тестовую карусель PNG.
7. `core/stories.py` — перенести `generate_stories.py`. Тест: PNG 9:16.
8. `core/stickers.py` — перенести `visual_engine.py`.
9. `posters/instagram.py` — перенести Postmypost. Тест опционален (требует токен).

После каждого подмодуля: один `python -c "from agents.agent4_publisher.core.X import ..."` для проверки импортов.

**Критерий приёмки:** все 9 подмодулей импортируются без ошибок, базовые тесты проходят.

### Этап 3 — Адаптация под студию (1 сессия, ~2 часа)

1. В `core/image.py` заменить промпты с киберпанка на архитектурные (см. §6).
2. В `core/carousel.py` заменить логотип / цветовую схему на студийную.
3. Подключить системный промпт студии в `core/llm.py`: загружать из `content/studio_brand/system_prompt.md`.
4. Подключить базу страхов из `content/studio_brand/audience_fears.md`.
5. Реализовать `cli.py`: `generate <kind> --topic ...` и `publish <channel> --file ...`.
6. Тестовый прогон: `.venv/bin/python -m agents.agent4_publisher.cli generate post --topic "Что такое инсоляция" --dry-run` → текст в `output/`.

**Критерий приёмки:** dry-run генерирует контент, который выглядит как пост студии, а не курса. Изображение — архитектурное.

### Этап 4 — Подключение к Redis-bus (1 сессия, ~1 час)

1. Реализовать `core/event_bus.py`: функция `publish_content_event(channel, post_url, ...)` → `redis.publish("content_published", json.dumps(...))`.
2. В `posters/telegram.py` после успешной публикации вызывать `publish_content_event`.
3. В `agents/agent5_crm/` (или отдельный subscriber) подписаться на `content_published` и логировать в `logs/`.
4. Тест: реальная публикация в тестовый Telegram-канал → событие в Redis → лог в Agent 5.

**Критерий приёмки:** опубликованный пост → Redis событие → запись в логах CRM.

### Этап 5 — Боевая публикация и архив `sales_ai` (1 сессия, ~30 мин)

1. Один реальный пост в Telegram-канал студии ВПП через `cli.py publish`.
2. Проверить: пост опубликован, событие в Redis, Agent 5 видит публикацию.
3. Обновить `REPORT.md` lead-engine: записать дату миграции, что переехало, что выкинуто.
4. Удалить `INTEGRATION.md` из бывшего `sales_ai/` (информация уже в новом `REPORT.md`).
5. Переименовать `projects/sales_ai/` → `projects/_archive_sales_ai_2026-05/` (или удалить, если git-история сохранена в Codex).
6. Удалить запись о `sales_ai` в `~/.claude/CLAUDE.md` (раздел «Файловая структура») — пройтись по упоминаниям.
7. Удалить запись в `~/.claude/knowledge_base.md`.

**Критерий приёмки:** пост опубликован, `sales_ai/` отсутствует в активных проектах, документация обновлена.
