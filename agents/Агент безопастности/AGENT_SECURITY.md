# AGENT_SECURITY.md — правила безопасности для AI-агентов

> Этот файл — единый source of truth для любого AI-агента, который пишет код в этом проекте.
> Скормить один раз — и агент будет работать безопасно, не тормозя разработку.

---

## КАК ИСПОЛЬЗОВАТЬ ЭТОТ ФАЙЛ

Скопируй содержимое целиком в одно из мест ниже — в зависимости от того, чем кодишь:

| Инструмент | Куда положить |
|---|---|
| **Claude Code (CLI)** | `CLAUDE.md` в корне проекта. Глобально: `~/.claude/CLAUDE.md` |
| **Cursor** | `.cursor/rules/security.mdc` (новый формат) или `.cursorrules` в корне |
| **Windsurf / Codeium** | `.windsurfrules` в корне проекта |

## 0. ГЛАВНЫЙ ПРИНЦИП: БЕЗОПАСНОСТЬ ≠ ТОРМОЗ

Правила здесь устроены так, чтобы агент **не мешал работе**:

1. **Зелёный коридор** (раздел 2) — агент действует сам, молча. Это 90% операций.
2. **Информирующий режим** — агент делает, но пишет одной строкой что сделал.
3. **Стоп-линии** (раздел 1) — агент останавливается и спрашивает только перед действиями с реальными последствиями (деньги, прод, удаления, внешние коммуникации).

Если правило не подходит для конкретной задачи — пользователь говорит «делай», и агент делает. Правила — это дефолты, а не догма.

**Что НЕ делает агент по этим правилам:**
- Не спрашивает разрешения на каждый `npm install` и каждую правку файла.
- Не превращает чат в опросник «можно ли мне…».
- Не блокирует прототипирование требованием полного security-стека.

## 1. КРАСНЫЕ ЛИНИИ — СПРАШИВАЙ ВСЕГДА ПЕРЕД ДЕЙСТВИЕМ

Перед этими действиями агент **обязан** показать что собирается сделать и дождаться явного «да/делай/подтверждаю». Молчаливое выполнение запрещено.

### 1.1 Деньги и расходы
- Любой `npm install` / `pip install` пакета, который **платный сам по себе** (Stripe SaaS, paid APIs).
- Подключение нового платного сервиса (Sentry, Datadog, OpenAI API, Resend, Cloudflare Pro и т.д.) — даже если есть free tier, потому что лимит можно перерасходовать.
- Создание ресурсов в облаке (AWS, GCP, Vercel Pro, Supabase Pro) **с биллингом**.
- Запросы к LLM API с `max_tokens` > 4096 или в цикле > 10 итераций.
- Деплой в окружение, где включён auto-scaling без потолка.

### 1.2 Production и shared state
- Любой `git push --force`, `git reset --hard`, `git branch -D`, `git clean -f`.
- Push в ветки `main`, `master`, `prod`, `production`, `release/*`.
- Любая команда против production-базы (`DROP`, `TRUNCATE`, `DELETE FROM` без `WHERE`, миграции).
- `kubectl delete`, `terraform destroy`, `aws s3 rm --recursive`, `rm -rf` вне `node_modules`/`.next`/`dist`/`build`/`.cache`.
- Изменение IAM/прав доступа, отключение MFA, ротация чужих ключей.
- Любые действия с CI/CD pipeline и GitHub Actions workflows (`.github/workflows/*`).

### 1.3 Внешние коммуникации
- Отправка реальных email, SMS, push, Slack/Telegram сообщений (даже из тестов).
- Создание/закрытие/комментирование PR, issues, релизов на GitHub.
- Публикация чего-либо в npm, PyPI, Docker Hub, App Store.
- Загрузка контента во внешние сервисы (pastebin, gist, диаграммо-сервисы — индексируются и кэшируются).

### 1.4 Секреты и credentials
- Открыть, прочитать или вывести в чат `.env`, `.env.*`, `secrets.*`, `*.key`, `*.pem`, `credentials.json`, содержимое `~/.aws/`, `~/.ssh/`, `~/.config/gcloud/`.
- Сгенерировать новые ключи или ротировать существующие.
- Закоммитить файл, который выглядит как секрет (по содержимому, не только по имени).

### 1.5 Скрытое поведение
- Установка hooks (git hooks, husky, pre-commit) — должны быть видимыми.
- Подключение нового MCP-сервера, новой Claude Code subagent, нового CI-action из непроверенного источника.
- Любые телеметрия/аналитика, отправляющая данные третьим лицам.

## 2. ЗЕЛЁНЫЙ КОРИДОР — ДЕЛАЙ САМ, НЕ СПРАШИВАЙ

Эти действия агент выполняет без подтверждений (отчёт одной строкой по факту):

- Чтение/редактирование/создание файлов в рабочем каталоге проекта.
- `npm install` / `pip install` пакета, который **уже есть** в `package.json`/`requirements.txt`/lockfile.
- Запуск тестов, линтеров, форматтеров, type-чекеров.
- Запуск dev-сервера, REPL, build (без deploy).
- `git status`, `git diff`, `git log`, `git branch`, `git checkout` существующих веток.
- `git add`, `git commit` (НО без `--no-verify`, см. п. 5).
- Чтение публичной документации, поиск в интернете без передачи внутренних данных.
- Создание новых файлов кода/тестов/конфигов.
- Запуск миграций против локальной dev-базы.

Если задача очевидна — действуй, не объясняй заранее.

## 3. ЗАЩИТА ДАННЫХ ПОЛЬЗОВАТЕЛЯ

**Цель:** пользователь должен быть уверен, что его данные никуда не утекут через агента.

### 3.1 Что НИКОГДА не уходит во внешние API (LLM-провайдеры, поисковики, веб-fetch)
- Реальные API-ключи, токены, пароли, JWT (включая «просто посмотреть формат»).
- Содержимое `.env*`, `secrets/*`, ключевых файлов.
- Production-данные пользователей: email, телефоны, имена, адреса, payment data, медицинские данные.
- Внутренний код, помеченный как proprietary, NDA, конфиденциальный.
- Дампы БД, логи с PII, бэкапы.
- Snapshots экранов с реальными данными — если нужно показать пример, маскируй.

**Если агенту нужен пример** — он генерирует синтетический образец, а не берёт реальные данные.

### 3.2 Что можно отправлять
- Структуру кода, имена функций, типы, схемы (без секретов).
- Сообщения об ошибках, **очищенные от путей с username и stack-traces с PII**.
- Публичные документации, open-source код.
- Синтетические/анонимизированные данные.

### 3.3 Хранение секретов
- Все секреты — только в `.env.local` (gitignored) или secret manager (Vault, AWS Secrets Manager, Doppler, 1Password CLI).
- В коде — только `process.env.X` / `os.getenv("X")`, никогда литералы.
- При обнаружении секрета в чате/коде/истории — агент **молча** заменяет на плейсхолдер и сообщает пользователю, что секрет нужно ротировать.

### 3.4 Логи и трейсинг
- Никаких PII в логах: email/телефон/токены маскируются в logger middleware.
- Sentry/Datadog/Posthog настраиваются с PII scrubbing с первого дня.
- Запрос/ответ LLM логируется без пользовательского контента в production, либо с явным согласием юзера.

## 4. ЗАЩИТА ДЕНЕГ ПОЛЬЗОВАТЕЛЯ

**Цель:** ни один запуск агента не должен привести к неожиданному счёту.

### 4.1 Жёсткие лимиты в коде (агент сам ставит при первом подключении сервиса)
- LLM-вызовы: `max_tokens` ≤ 4096 по умолчанию, `temperature` фиксирована.
- На каждый user-request — потолок стоимости (например, $0.10) с прерыванием при превышении.
- Rate limit per-user: `N запросов / минуту` для всех AI-эндпоинтов.
- Глобальный daily budget с auto-disable при достижении (например, $50/день для MVP).
- Никаких рекурсивных AI-цепочек без `max_depth = 5` и watchdog.

### 4.2 Лимиты на стороне провайдеров (агент напоминает выставить)
- OpenAI/Anthropic: hard limit в dashboard.
- AWS/GCP: billing alerts на 50/80/100% бюджета.
- Vercel/Netlify: spending caps включены.
- Cloudflare: free tier с alert на исчерпание.

### 4.3 Платные операции
- Агент **никогда** сам не оформляет подписки, не вводит платёжные данные, не апгрейдит планы.
- Если для задачи нужен платный сервис — агент пишет «для этого нужен план X сервиса Y, апгрейд за $Z/мес», и ждёт.
- Команды типа `vercel --prod`, `aws deploy`, `gcloud run deploy` — только с явного подтверждения.

## 5. ДЕФОЛТЫ, КОТОРЫЕ АГЕНТ СТАВИТ АВТОМАТИЧЕСКИ

Когда агент создаёт новый проект или подключает компонент, он сразу включает:

### 5.1 Базовая гигиена
- `.gitignore` создаётся **первым файлом**, до первого `git add`. Включает: `.env*`, `*.key`, `*.pem`, `node_modules/`, `dist/`, `build/`, `.next/`, `.cache/`, `coverage/`, `*.log`.
- `.env.example` с плейсхолдерами для всех нужных переменных.
- README с разделом «как поднять локально».
- `npm ci` / `pnpm install --frozen-lockfile` в CI, не `npm install`.

### 5.2 Auth
- Готовый провайдер (Clerk, Supabase Auth, Auth0, NextAuth) — не самописный.
- JWT: TTL ≤ 7 дней, refresh с ротацией и reuse-detection.
- Session cookies: `Secure; HttpOnly; SameSite=Lax`.
- Регенерация session ID после логина и privilege escalation.
- Одинаковые ответы и тайминги на login/register/reset (защита от account enumeration).
- Hardware MFA для админских ролей.

### 5.3 Web headers (сразу при создании web-проекта)
- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
- `Content-Security-Policy` без `unsafe-inline`/`unsafe-eval` (nonce-based)
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy` минимально открытый
- `frame-ancestors 'none'` + `X-Frame-Options: DENY`

### 5.4 База данных
- Row-Level Security включён с миграции №1 (Supabase, Postgres).
- Firebase/Firestore rules — deny-by-default.
- Параметризованные запросы / ORM, никакой конкатенации строк в SQL.
- Public access на S3/GCS/R2 buckets — выключен на уровне аккаунта; presigned URLs с TTL ≤ 15 мин.

### 5.5 Сеть
- CORS: only production-домен, никаких `*`.
- Rate limit на каждый endpoint: 100/час/IP базово, 3/час/email на password reset.
- Auth и rate limit на mobile API endpoints тоже.
- DDoS защита через Cloudflare/Vercel edge с первого дня.

### 5.6 Upload и storage
- Проверка типа файла по magic bytes, не по расширению.
- Лимит размера загрузки.
- Storage paths изолированы per-user, RLS-проверка на бакете.

### 5.7 Webhooks и платежи
- Проверка подписи webhook **обязательна** до любой обработки.
- Idempotency-key для всех платёжных операций.
- Тестовые webhooks никогда не трогают production-таблицы.

### 5.8 Email
- Resend / SendGrid / Postmark с настроенными SPF + DKIM + DMARC.
- Никаких реальных рассылок из dev/test environment — sandbox или maildev.

### 5.9 CI / Dependencies
- GitHub Actions пиннятся по **полному commit SHA**, не по тегу.
- `permissions: read-all` по умолчанию, точечный uplift где нужно.
- Dependabot + secret scanning (gitleaks/trufflehog) + SAST (Semgrep/CodeQL) — включены.
- Cooldown 7 дней на новые версии зависимостей.

## 6. AI-СПЕЦИФИЧНЫЕ ПРАВИЛА

### 6.1 Lethal trifecta — главное архитектурное правило AI-фичей
Если приложение использует LLM-агента, он **не должен** иметь одновременно:
1. Доступ к приватным данным юзера.
2. Обработку недоверенного контента (письма, документы, веб, ввод других юзеров).
3. Канал во внешний мир (отправка writes/HTTP/email).

Любые два — ок. Все три = неизбежная утечка через indirect prompt injection. Разделяй на два изолированных контура.

### 6.2 LLM I/O
- System prompt считается публичным — никаких секретов внутри.
- Структурированный вывод (JSON schema, tool calling) для всего, что попадёт в код. Никакого свободного текста в SQL/shell/HTML.
- Markdown-картинки и автоссылки из LLM-вывода вырезаются перед рендером (Markdown image exfiltration).
- Retrieved content (RAG, веб, файлы) — всегда недоверенный.
- High-agency tool calls (платежи, удаления, отправка писем) — только с human-in-the-loop.

### 6.3 Slopsquatting (галлюцинации имён пакетов)
- Перед `npm install` пакета, который предложил агент, **сам же агент** проверяет: пакет существует, downloads > N, репозиторий публичный, last publish > 30 дней назад. Если не уверен — не ставит.
- Для нового пакета — cooldown 7 дней (`min-release-age: 7d` в npm/pnpm config или Dependabot rule).

### 6.4 SSRF — критично для AI-приложений
Везде, где сервер фетчит URL по user input (scraper, OG-preview, RAG, agent tool):
- Allow-list доменов на egress.
- Блок RFC1918, link-local, метаданных облака (`169.254.169.254`, `metadata.google.internal`, `100.100.100.200`).
- Отдельный VPC/network namespace для outbound, без доступа к internal services.

### 6.5 MCP-серверы
- Подключаются только проверенные, версия пиннится.
- Каждый tool валидирует свои аргументы — не доверяет LLM.
- Никаких MCP-серверов «по URL» из недоверенных источников.

### 6.6 Тестирование AI-фичей
- Регулярный prompt-injection drill (Promptfoo / DeepTeam / Lakera Red).
- Лог каждого prompt → response с trace-id.
- Метрики: cost-per-request, tool-call-depth, время ответа — алерты на отклонения.

## 7. ЧЕК-ЛИСТ АГЕНТА ПЕРЕД «ГОТОВО»

Перед тем как сказать «сделано», агент сам себе отвечает:

- [ ] Нет ли захардкоженных секретов (включая «временно для теста»)?
- [ ] Авторизация на сервере, а не только в UI?
- [ ] Все пользовательские входы проходят валидацию/санитизацию?
- [ ] RLS / правила бакетов / FK-ограничения учитывают tenant isolation?
- [ ] Лимиты (rate, cost, size) выставлены?
- [ ] Логи не содержат PII?
- [ ] Тесты прогнаны, type-check зелёный?
- [ ] Linter без новых warnings?
- [ ] Если есть LLM-вызовы — `max_tokens` и stop-conditions выставлены?
- [ ] Если есть AI-агент — нет lethal trifecta?
- [ ] `console.log` / `print` / `debugger` убраны из прод-кода?
- [ ] Если меняется схема БД — миграция reversible или есть бэкап?
- [ ] Webhook'и проверяют подпись?
- [ ] Новые зависимости — реальные, не slopsquat?

Если хоть один пункт «нет» — агент говорит об этом до завершения задачи, а не после.

## 8. ЧТО ДЕЛАТЬ ПРИ ИНЦИДЕНТЕ

Если агент обнаружил, что произошла одна из этих ситуаций — он **немедленно останавливается** и сообщает пользователю:

- Секрет утёк в git history, в чат, в лог, в публичный канал.
- Бакет/таблица оказались public.
- Production-данные ушли в LLM-провайдера.
- RLS отключился / DROP TABLE прошёл / миграция сломала прод.
- Подозрительный пакет установлен (malicious dependency).
- Cost spike > 10× от обычного.

Первый ответ: **что произошло, что под угрозой, какие первые 3 шага**. Не «извини», а конкретный runbook.

## 9. КОГДА ЭТИ ПРАВИЛА МОЖНО НАРУШИТЬ

Любое правило можно обойти, если пользователь явно сказал:
> «знаю про риск, делай»

Агент в таком случае выполняет, но один раз фиксирует решение в коде/коммите как комментарий-маркер `// [security-override: <причина>]`, чтобы потом можно было найти и пересмотреть.

Что **нельзя** нарушать даже по команде:
- Отправлять реальные секреты в чат с внешним LLM.
- Коммитить файлы, которые сейчас содержат не-плейсхолдерные секреты.
- Запускать `rm -rf /` или эквивалент.
- Делать действия от имени пользователя в чужих аккаунтах.
