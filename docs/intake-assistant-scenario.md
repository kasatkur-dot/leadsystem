# Adaptive Intake Assistant Scenario

Дата: 2026-05-14

Проект: `design-studio-lead-engine`

Статус: архитектурный сценарий. Код, внешние сервисы, платные API, реальные клиенты и публикации не запускались.

## 1. Главный вывод

Идея AI-ассистента из примера El Remont нам подходит, но для Вектор Плюс-Про её нельзя делать как жёсткую анкету.

Правильная версия для проектной организации:

```text
умный диалоговый ассистент
-> мягко понимает задачу
-> собирает обязательные факты
-> не обещает невозможного
-> создаёт карточку обращения / сделки
-> передаёт в Agent 3 и Agent 5
```

То есть ассистент не должен идти строго так:

```text
вопрос 1 -> вопрос 2 -> вопрос 3 -> вопрос 4
```

Он должен работать так:

```text
человек пишет как умеет
-> ассистент понимает смысл
-> видит, каких фактов не хватает
-> задаёт 1-2 уместных уточнения
-> собирает карточку лида
```

## 2. Почему не жёсткие вопросы

Жёсткая анкета плоха для нашей ниши, потому что люди приходят с разными задачами:

- `Сколько стоит проект перепланировки?`
- `Нужен КР для реконструкции`
- `Есть коммерческое помещение, нужно понять, можно ли изменить вход`
- `Нужна РД после стадии П`
- `Нужен проект для экспертизы`
- `Есть старые чертежи, надо проверить`

Если всем задавать одинаковые 7 вопросов, диалог будет деревянным.

Лучше: модель ведёт разговор свободно, но внутри держит список обязательных фактов.

## 3. Что значит “умная модель” в нашем проекте

Умная модель — это не магия и не “сама всему научилась”.

Безопасная схема:

```text
LLM-модель
+ системная инструкция ВПП
+ база знаний проекта
+ правила запретов
+ обязательные поля лида
+ примеры хороших ответов
+ ручное ревью первых диалогов
```

Важно: модель не должна самостоятельно обучаться на закрытых клиентских данных без отдельного решения и правил безопасности.

На MVP нам достаточно:

- хорошей инструкции;
- базы знаний из наших документов;
- обязательного JSON/карточки лида на выходе;
- dry-run тестов на синтетических диалогах;
- ручного ревью Яники.

Fine-tuning, векторная база, RAG, OpenRouter/OpenAI/Anthropic и полноценная память диалогов — это позже, после безопасного теста.

## 4. Обязательные факты, а не обязательные вопросы

Ассистент должен собрать не “ответы на 7 вопросов”, а факты:

| Факт | Зачем нужен | Пример |
|---|---|---|
| `city` | Понять регион и правила | Краснодар, Сочи, Москва |
| `object_type` | Понять тип объекта | квартира, коммерческое помещение, здание, реконструкция |
| `task_type` | Понять услугу | ПД, РД, КР, КЖ, КМ, перепланировка, реконструкция |
| `area_m2` | Прикинуть масштаб | 70 м2, 250 м2, 1200 м2 |
| `existing_docs` | Понять исходные данные | план БТИ, эскиз, ГПЗУ, старый проект, фото |
| `timeline` | Понять срочность | срочно, в течение месяца, планово |
| `contact` | Куда вернуться с ответом | телефон, email, MAX, Telegram |
| `budget_hint` | Понять готовность и ожидания | известен, не определён, есть ориентир |
| `next_step_preference` | Что удобно человеку | консультация, КП, звонок, список документов |

Если человек уже написал часть информации, ассистент не спрашивает это заново.

## 5. Как ассистент должен вести диалог

### Разрешённая логика

Ассистент может:

- встретить человека коротко и спокойно;
- объяснить, что для точного ответа нужны исходные данные;
- задать 1-2 уточняющих вопроса за раз;
- дать предварительное направление, а не точную смету;
- предложить прислать план, эскиз или описание;
- аккуратно передать заявку специалисту;
- сформировать карточку лида для Bitrix24.

### Нежелательная логика

Ассистент не должен:

- превращать диалог в длинную анкету;
- требовать телефон до базового ответа;
- давать точную цену без исходных данных;
- обещать согласование;
- говорить, что проектная организация сама принимает решение о согласовании;
- спорить с человеком;
- давать юридическое заключение без проверки;
- собирать лишние персональные данные.

## 6. Что можно отвечать

Пример хорошего ответа:

```text
Поняла задачу. Для первого разбора не нужен полный комплект документов.

Чтобы подсказать следующий шаг, уточните, пожалуйста:
1. В каком городе объект?
2. Это квартира, коммерческое помещение, здание или реконструкция?
3. Есть ли план, эскиз, БТИ или старые чертежи?

После этого можно будет понять, какие исходные данные нужны и какой формат работы подойдёт: консультация, состав проекта, ПД/РД или конструктивные разделы.
```

Если человек спрашивает цену:

```text
Стоимость зависит от типа объекта, площади, состава разделов и исходных документов.

Я могу предварительно сориентировать по составу работ, но для точной оценки нужно понять:
- город;
- тип объекта;
- площадь;
- что именно нужно сделать;
- какие документы уже есть.
```

Если человек пишет ночью:

```text
Спасибо, я зафиксирую задачу и передам специалисту.

Напишите, пожалуйста, город, тип объекта и что нужно сделать. Если есть план или эскиз — его можно прислать позже.
```

## 7. Что нельзя обещать

Нельзя писать:

- `100% согласуем`;
- `гарантированное согласование`;
- `узаконим любую перепланировку`;
- `точная цена сразу`;
- `сделаем без документов`;
- `обойдём требования`;
- `решение будет точно положительным`;
- `проектная организация согласует сама`.

Правильная формулировка:

```text
Мы можем подготовить проектную документацию, проверить исходные данные и подсказать безопасный следующий шаг. Решение о согласовании принимает уполномоченный орган.
```

## 8. Карточка лида на выходе

Ассистент должен отдавать карточку, которую потом можно передать Agent 3 и Agent 5.

Минимальный формат:

```text
lead_source: site_ai_assistant / max_ai_assistant / telegram_ai_assistant / bitrix_openline
city:
object_type:
task_type:
area_m2:
existing_docs:
timeline:
budget_hint:
contact:
contact_channel:
raw_dialog_summary:
missing_facts:
lead_segment:
score_hint:
recommended_next_step:
consent_status:
```

Как это связано с текущими моделями:

| Поле ассистента | Куда потом ложится |
|---|---|
| `lead_source` | `RawLead.source` |
| `city` | `RawLead.city`, `QualifiedLead.city` |
| `object_type` | `QualifiedLead.object_type` |
| `area_m2` | `QualifiedLead.area_m2` |
| `raw_dialog_summary` | `RawLead.raw_text` |
| `contact` | `RawLead.contact`, `QualifiedLead.contact` |
| `lead_segment` | будущая CRM/аналитика, сейчас можно хранить в тексте/метаданных |
| `recommended_next_step` | `QualifiedLead.recommended_action` |
| `consent_status` | `RawLead.consent_status`, `QualifiedLead.consent_status` |

## 9. Как это входит в нашу многоагентную систему

```text
Сайт / MAX / Telegram / Bitrix Open Lines
-> Adaptive Intake Assistant
-> RawLead / карточка входа
-> Agent 3 Processor
-> score + offer_text + recommended_action
-> Agent 5 CRM
-> Bitrix24 lead/deal/task
-> уведомление человеку внутри ВПП
-> аналитика источников
```

Роли:

| Блок | Ответственность |
|---|---|
| Agent 4 Publisher | тексты, сценарий общения, база знаний, FAQ, tone of voice |
| Agent 3 Processor | квалификация, скоринг, следующий шаг |
| Agent 5 CRM | сделка в Bitrix24, задача, уведомление, аналитика |
| Agent 6 Outreach | позже: безопасное первое касание в чатах после approval |
| Человек | ревью первых диалогов, спорные случаи, решение по отправке КП |

## 9.1. Связь с AI-менеджером сделки

После уточнения от 2026-05-15 ассистент рассматривается не только как сборщик фактов, а как AI-менеджер первого касания.

Цель:

```text
AI отвечает как менеджер ВПП
-> ведёт диалог до ясного следующего шага
-> создаёт/обновляет сделку
-> передаёт человеку, если нужны цена, договор, сроки или техническое решение
```

Подробное правило вынесено отдельно:

```text
docs/ai-deal-manager-responsibility.md
```

## 10. Чем это отличается от жёсткого бота

Жёсткий бот:

```text
Задаёт одинаковые вопросы всем.
```

Наш ассистент:

```text
Понимает смысл сообщения.
Смотрит, какие факты уже есть.
Спрашивает только то, чего не хватает.
Заполняет карточку лида.
Не обещает лишнего.
Передаёт человеку понятный следующий шаг.
```

## 11. Что нужно для реализации позже

Для первого MVP без внешних API:

- 10-15 синтетических диалогов;
- ручная проверка, какие поля ассистент должен извлечь;
- шаблон карточки лида;
- правила запретов;
- эталонные ответы.

Для реальной версии:

- выбранная LLM-модель;
- системная инструкция;
- база знаний ВПП;
- канал подключения: сайт / Bitrix Open Lines / MAX / Telegram;
- логирование диалогов без секретов;
- ручное ревью первых 72 часов;
- правило передачи горячих лидов человеку.

## 12. Маленький тест без API

Первый тест не должен запускать внешнюю модель.

Тест:

```text
3 синтетических сообщения клиента
-> вручную заполнить expected intake card
-> проверить, хватает ли полей для Agent 3 и Agent 5
```

Пример сообщения:

```text
Здравствуйте. Нужно понять, сколько будет стоить проект для реконструкции коммерческого помещения в Краснодаре, площадь примерно 180 м2. Есть старый план и фото.
```

Ожидаемая карточка:

```text
lead_source: manual_fixture
city: Краснодар
object_type: коммерческое помещение
task_type: реконструкция / проектная документация
area_m2: 180
existing_docs: старый план, фото
timeline: unknown
budget_hint: unknown
contact: unknown
missing_facts: сроки, контакт, точный состав работ
lead_segment: commercial_nonresidential
score_hint: warm
recommended_next_step: запросить план/фото и предложить короткий разбор исходных данных
consent_status: unknown
```

## 13. Следующий маленький шаг

Создать тестовый документ с 3 синтетическими диалогами и expected intake card.

Код не менять, Bitrix24 не запускать, реальные клиенты не использовать.
