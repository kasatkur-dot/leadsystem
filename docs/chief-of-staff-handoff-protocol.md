# Chief of Staff and Handoff Protocol

Дата: 2026-05-10

Статус: точечное усиление текущей 6-агентной системы по мотивам внешнего референса `petukhova2023-bit/dz-ai-day-5`. Код и чужие файлы не копировались. Новые агенты не создавались.

## 1. Что берём из референса

Полезные принципы:

- координатор / Chief of Staff;
- нервная система: вход задачи -> маршрутизация -> агенты -> QA -> результат -> память;
- единый task-handoff;
- единый agent-result;
- escalation-to-yana;
- weekly digest;
- правило: агент = роль, skill = действие.

Что не берём:

- не расширяем систему до 26 агентов;
- не переносим чужие имена агентов;
- не копируем шаблоны дословно;
- не меняем MVP-архитектуру `6 агентов -> Redis -> Agent 5 -> Bitrix24/Telegram/Reports`.

## 2. Как это ложится на наш проект

В нашем проекте Chief of Staff - это не новый Agent 7. Это режим управления внутри существующей верхней логики:

```text
CLAUDE.md / AGENTS.md -> правила и роль координатора
REPORT.md -> память
docs/* -> карта и протоколы
scripts/check_agent_okr_contract.py -> QA/checker
Agent 1-6 -> рабочие департаменты
Agent 5 -> финальная CRM/аналитика/weekly digest позже
```

Простыми словами: координатор не делает работу за агентов. Он принимает задачу, выбирает нужный маршрут, проверяет результат и записывает память.

## 3. Нервная система для lead-engine

```text
Вход задачи
-> восстановить контекст из REPORT.md и правил
-> определить тип задачи
-> выбрать 1-3 нужных агента, не всех сразу
-> оформить handoff
-> получить agent-result
-> проверить QA/checker
-> если есть риск, сделать escalation-to-yana
-> отдать результат
-> обновить REPORT.md / нужный docs-файл
```

### Типовые маршруты

| Тип задачи | Маршрут | Итог |
|---|---|---|
| Новый лид | Agent 2 -> Agent 3 -> Agent 5 | Bitrix24 лид, Telegram уведомление, отчёт |
| Новый источник | Agent 1 -> Agent 2 -> Agent 5 | source card, тест одного источника, канал в аналитике |
| Контент/доверие | Agent 1 -> Agent 4 -> Agent 5 | черновик, approval, future content event |
| Outreach | Agent 6 -> approval человека -> Agent 5 | безопасный ответ, interested lead |
| Проверка системы | checker -> dashboard -> REPORT.md | статус файлов, OKR, метрик, блокеров |

## 4. Task-handoff

Использовать, когда один агент или слой передаёт задачу другому.

```text
От кого:
Кому:
Цель:
Контекст:
Что уже известно:
Что нужно сделать:
Формат результата:
Куда положить результат:
Срок / условие готовности:
Риски и ограничения:
Нужна ли Яника для решения: да/нет
```

Минимальные правила:

- handoff должен быть коротким;
- в нём должен быть один ожидаемый результат;
- если есть риск денег, публикации, персональных данных или юридического обещания - сразу ставить `Нужна Яника: да`;
- не передавать задачу всем агентам сразу.

## 5. Agent-result

Использовать после работы агента или тестового скрипта.

```text
Кто выполнял:
Что сделано:
Что проверено:
Файлы / отчёты:
Статус: OK / FAILED / BLOCKED / DRY_RUN / NOT_RUN
Ключевые выводы:
Риски:
Что нужно от другого агента:
Что нужно от Яники:
Следующий маленький шаг:
```

Для runtime-тестов обязательны статусы внешних вызовов:

```text
redis=True/False
bitrix24=True/False
telegram_send=True/False
imap=True/False
llm=True/False
scheduler=True/False
publisher=True/False
```

## 6. Escalation-to-yana

Эскалация нужна не для всего подряд, а только когда без человека есть риск принять неправильное решение.

Обязательно эскалировать:

- запуск платного API или рост бюджета;
- реальную публикацию;
- массовый сбор лидов;
- реальную outreach-отправку;
- изменение CRM-логики;
- удаление или перенос файлов;
- юридические обещания клиентам;
- персональные данные;
- выбор между двумя вариантами с разными последствиями.

Формат:

```text
Ситуация:
Почему нельзя решить автоматически:
Вариант 1:
Плюсы:
Минусы:
Вариант 2:
Плюсы:
Минусы:
Рекомендация координатора:
Что нужно решить Янике:
```

## 7. Weekly digest

Weekly digest - это будущий отчётный слой, а не новый агент.

Ответственный слой: Agent 5 CRM/Analytics + dashboard reports.

Периодичность: позже, 1 раз в неделю, после стабильного MVP.

Формат:

```text
Период:
Главный фокус недели:
Что сделано:
Что проверено:
Лиды и CRM:
Контент и доверие:
Outreach:
ROMI / каналы:
Блокеры:
Риски:
Решения, которые нужны от Яники:
План на следующую неделю:
```

Что важно: weekly digest должен собираться из `REPORT.md`, `data/reports/*.json`, dashboard и CRM-статусов. Нельзя писать туда секреты, токены, webhook URL и персональные данные.

## 8. Правило agent = role, skill = action

В нашей системе:

```text
Агент = роль / департамент / ответственность.
Skill = конкретное действие, которое помогает роли.
```

Примеры:

| Агент | Роль | Skill как действие |
|---|---|---|
| Agent 1 Scout | разведка источников | найти source card, сравнить конкурентов, выделить боли |
| Agent 2 Collector | сбор лидов | прочитать письмо, нормализовать источник, создать RawLead |
| Agent 3 Processor | квалификация | очистить, обогатить, скорить, сделать оффер |
| Agent 4 Publisher | контент и доверие | написать пост, подготовить визуал, создать content event |
| Agent 5 CRM | CRM и аналитика | создать лид, уведомить менеджера, посчитать ROMI |
| Agent 6 Outreach | первое касание | найти обсуждение, подготовить ответ, дождаться approval |

Вывод: если нужен новый навык, мы добавляем skill к существующему агенту. Нового агента создаём только если появляется новая зона ответственности, а сейчас это запрещено.

## 9. 3 точечных улучшения для проекта

1. Зафиксировать Chief of Staff как режим управления в существующей системе, а не как нового агента.
2. Использовать единый формат `task-handoff` и `agent-result` в документах, отчётах и будущей панели.
3. Добавить weekly digest как будущий блок Agent 5/dashboard после стабильного MVP.
