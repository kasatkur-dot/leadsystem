## 1. Состав архива

В архиве `11` файлов:

- `index.html`;
- `app.jsx`;
- `data.js`;
- `drawings.jsx`;
- `image-slot.js`;
- `layout.jsx`;
- `page-home.jsx`;
- `page-inner.jsx`;
- `page-extras.jsx`;
- `styles.css`;
- `tweaks-panel.jsx`.

Это не готовый production-проект, а статический React-макет, который запускается через CDN-версии `React`, `ReactDOM` и `Babel` прямо в браузере.

## 2. Что сделано хорошо

- Структура сайта в целом соответствует нашему ТЗ: главная, услуги, страницы услуг, кейсы, `О компании`, `СРО`, контакты, медиа/social preview.
- Старый Bitrix24-сайт не используется как будущая публичная ссылка.
- Важные данные компании вынесены в текст: `ООО «Вектор Плюс-Про»`, `ИНН`, `ОГРН`, ссылки на Яндекс Карты, 2GIS, СРО и канал.
- Есть аккуратные формулировки без обещания “100% согласуем”.
- Есть визуальная форма, но она не отправляет данные: `onSubmit={(e) => e.preventDefault()}`.
- Секреты, токены, webhook, Bitrix24, Supabase, OpenRouter, Anthropic, Telegram API в коде не найдены.
- Медиа-слой учтён: есть промо-блок, схема пути клиента, social preview, вертикальные кадры для MAX/Telegram/VK/Дзена.
- Автозвук не найден; промо-блок явно говорит, что звук только при ручном запуске будущего ролика.

## 3. Что не готово к публикации

### 3.1. Технический формат

Сейчас сайт зависит от внешних CDN:

```text
https://unpkg.com/react@18.3.1/umd/react.development.js
https://unpkg.com/react-dom@18.3.1/umd/react-dom.development.js
https://unpkg.com/@babel/standalone@7.29.0/babel.min.js
https://fonts.googleapis.com
https://fonts.gstatic.com
```

Проблема:

- это development-режим, не production;
- сайт зависит от внешнего интернета;
- браузер компилирует JSX на стороне клиента через Babel;
- для Vercel/нормального сайта лучше собрать проект через локальный build.

Решение:

```text
перенести макет в нормальную структуру сайта: Vite/React или Next.js,
убрать Babel из браузера,
собрать production bundle
```

### 3.2. Внутренние инструменты дизайнера

В коде есть:

- `tweaks-panel.jsx`;
- `image-slot.js`;
- `image-slot` в шапке сайта.

Это похоже на редакторские/дизайнерские инструменты для настройки макета и загрузки изображений.

Проблема:

- посетитель не должен видеть панель `Tweaks`;
- загрузчик изображений на стороне клиента не нужен в production;
- `image-slot.js` содержит логику drag/drop, localStorage/fetch sidecar и служебные контролы.

Решение:

```text
оставить эти элементы только как внутренний инструмент,
а в production заменить на обычные img/svg/components
```

### 3.3. Скрытый SEO/GEO-блок

В `index.html` есть блок:

```text
position:absolute; left:-9999px
```

Он содержит ключевые данные для AI/GEO.

Проблема:

- скрытый текст может выглядеть как SEO-переспам;
- лучше, чтобы важные данные были видимыми обычными блоками на сайте.

Решение:

```text
убрать скрытый блок или заменить его видимым блоком “Публичные данные компании”
```

### 3.4. Кейсы выглядят как реальные

В `data.js` есть карточки кейсов с годами и типами объектов:

- `Многоквартирный жилой дом`;
- `Коммерческий объект`;
- `Нежилое помещение`;
- `Производственное здание`;
- `Бюджетный объект`;
- `Частный жилой дом`.

Проблема:

- в ТЗ было правило не придумывать реальные кейсы;
- если эти кейсы не подтверждены, они должны быть явно помечены как шаблоны или заменены на реальные проверенные кейсы.

Решение:

```text
до публикации либо заменить на проверенные кейсы,
либо подписать как “пример структуры кейса / шаблон”
```

### 3.5. Контакты неполные

Сайт содержит ссылки на карты и канал, но нет нормального набора контактов:

- телефон;
- email;
- Telegram;
- адрес офиса текстом.

Решение:

```text
добавить только подтверждённые публичные контакты после ручной проверки
```

### 3.6. Форма может путать пользователя

Форма не отправляет данные, но кнопка называется:

```text
Отправить заявку
```

Проблема:

- человек может подумать, что заявка реально ушла.

Решение для preview:

```text
оставить как визуальную форму, но кнопку назвать “Заготовка формы”
или заменить на CTA “Написать в канал / открыть контакты”
```

Решение для production:

```text
подключать реальную отправку только позже,
после решения по CRM/Bitrix24/Agent 5
```

### 3.7. Визуальные ассеты не вставлены

В макете пока нет наших настоящих:

- логотипов;
- favicon;
- redacted-отзывов;
- redacted-документов;
- social preview-файлов;
- подготовленного hero-изображения.

Решение:

```text
сопоставить макет с docs/site-visual-assets-safe-review.md
и подготовить asset plan
```

## 4. Что можно использовать как основу

Можно использовать:

- структуру страниц;
- визуальный стиль;
- промо-блок без автозвука;
- схему пути клиента;
- social preview-страницу;
- аккуратные формулировки услуг;
- централизованный `data.js` как основу будущего контентного слоя.

Не использовать без переработки:

- `tweaks-panel.jsx`;
- `image-slot.js`;
- CDN React/Babel;
- скрытый SEO-блок;
- placeholder-кейсы как реальные;
- visual form как реальную заявку.

## 6. Рекомендованный следующий шаг

Следующий маленький шаг:

```text
не публиковать сайт;
создать локальную рабочую папку будущего сайта;
перенести туда макет как v1;
убрать/отложить production-блокеры:
CDN React/Babel, TweaksPanel, image-slot, скрытый SEO-блок;
затем подключить проверенные ассеты
```

Перед переносом в production нужно отдельно проверить:

- мобильную версию в браузере;
- доступность;
- скорость;
- ссылки;
- юридические формулировки;
- подтверждение кейсов;
- подтверждение канала `СИЛА Проекта`;
- подтверждение СРО и публичных контактов.
