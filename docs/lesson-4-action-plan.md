## 2. Безопасный GitHub-цикл

Цель GitHub-цикла: проект можно передать, восстановить на другом компьютере и продолжить без потери правил, структуры и кода. При этом секреты не должны попасть в репозиторий.

### Что уже хорошо

- Git remote настроен на `https://github.com/kasatkur-dot/leadsystem.git`.
- `.gitignore` закрывает `.env`, `.env.*`, `secrets/`, `credentials/`, Telethon-сессии, логи, `.venv`.
- Есть `README.md`, `CLAUDE.md`, `REPORT.md`, `AGENTS.md`.
- Есть `.env.example`, который показывает, какие ключи нужны, но не хранит реальные значения.

### Что обязательно делать перед каждым push

1. Проверить статус:

```bash
git status --short
```

2. Убедиться, что в список не попали секреты:

```bash
git status --short | grep -E "\\.env|session|secret|credential|\\.log|\\.jsonl" || true
```

3. Проверить, что `.env` не отслеживается Git:

```bash
git ls-files .env .env.example
```

Правильный результат: `.env.example` может быть в Git, `.env` не должен быть в Git.

4. Передать в GitHub только код, документы, шаблоны и безопасные CSV.

5. Не коммитить:

- `.env`;
- реальные Telegram `.session`;
- API-ключи;
- логи с персональными данными;
- выгрузки лидов с телефонами/email без отдельного решения;
- локальные временные видео/аудио/скриншоты.

### Как восстановить проект на новом компьютере

```bash
git clone https://github.com/kasatkur-dot/leadsystem.git
cd leadsystem
cp .env.example .env
```

Дальше человек вручную заполняет `.env`, устанавливает зависимости, запускает Redis, авторизует Telegram-сессии и проверяет систему по `REPORT.md` и `CLAUDE.md`.

### Что важно для восстановления

В GitHub должны лежать:

- код агентов;
- `README.md`;
- `CLAUDE.md`;
- `AGENTS.md`;
- `REPORT.md`;
- `docs/`;
- `research/`;
- `.env.example`;
- безопасные справочники каналов;
- dry-run инструкции.

В GitHub не должны лежать:

- реальные ключи;
- реальные сессии;
- реальные клиентские контакты без обезличивания;
- файлы, которые можно восстановить только на локальном компьютере.

## 3. Принцип “один источник сначала”

Сейчас нельзя одновременно запускать все платформы:

- Gmail;
- Telegram;
- Avito;
- Profi.ru;
- Яндекс Услуги;
- карты;
- Reels;
- Ads;
- SEO;
- Influencer-TG.

Так система станет большой, дорогой и трудной для отладки.

Правильный порядок:

```text
сначала один источник -> потом второй -> потом автоматизация -> потом масштабирование
```

### Где применять этот принцип в проекте

| Зона | Как делать правильно |
|---|---|
| Агент 2 / сбор лидов | Сначала один источник, например `tender_email` или ручной `avito`, не все collector-модули сразу |
| Агент 6 / аутрич | Сначала один канал Telegram, не Telegram + VK + форумы + TenChat одновременно |
| Агент 4 / контент | Сначала один канал публикации в dry-run, потом реальные площадки |
| Reels Radar | Сначала offline-список из 5-10 роликов, потом Apify |
| Сквозная аналитика | Сначала ручные CSV, потом PostgreSQL/Supabase |
| Ads | Сначала UTM и посадочная точка, потом бюджет |

### Рекомендованный первый источник для теста

Для MVP лучше начинать с самого контролируемого источника:

```text
tender_email -> Agent 2 -> Redis raw -> Agent 3 -> Redis qualified -> Agent 5 -> Bitrix/уведомление
```

Но сейчас этот путь заблокирован пустыми `.env` полями и Redis.

Если хотим тест без внешних доступов, лучше сделать учебный dry-run:

```text
ручной тестовый RawLead -> Agent 3 -> Agent 5 dry-run/лог -> отчёт
```

Это не стоит денег и позволяет проверить структуру.

## 4. Минимальный экономный тест Apify/OpenRouter

Платные действия сейчас не запускаем.

### Уровень 0 — без денег и без внешних API

1. Вручную взять 5 примеров источников или Reels.
2. Заполнить локальный CSV/fixture.
3. Посчитать базовые метрики.
4. Сгенерировать Markdown-отчёт.
5. Проверить, что Агент 4 может взять идею и сделать dry-run контент.

Это самый безопасный первый тест.

### Уровень 1 — OpenRouter после подтверждения

OpenRouter использовать только после отдельного подтверждения:

- какой ключ используется;
- какой лимит расходов;
- какая модель;
- сколько записей анализируем;
- какой максимум токенов.

Экономный тест:

```text
3 записи -> 1 короткий промпт на запись -> только текстовый анализ -> сохранить результат
```

Не использовать OpenRouter для массового анализа, пока нет лимита и отчёта стоимости.

### Уровень 2 — Apify после подтверждения

Apify-клиент может запускать Actor и читать dataset. Это удобно, но может стоить денег.

Экономный тест после подтверждения:

```text
1 Actor -> 1 источник -> limit 5-10 результатов -> metadata only -> без скачивания видео
```

Что запрещено без отдельного разрешения:

- массовый сбор;
- скачивание видео;
- анализ аудио;
- frame extraction;
- долго работающие Actors;
- автозапуск по расписанию.

## 5. Где хранить данные: Redis, PostgreSQL, Supabase

### Текущая архитектура

| Данные | Сейчас | Потом |
|---|---|---|
| Очереди лидов | Redis: `leads:raw`, `leads:qualified`, `leads:outreach` | Остаётся Redis |
| События контента | Redis: `content:published` | Дублировать в PostgreSQL/Supabase |
| Квалифицированные лиды | PostgreSQL таблица `leads` в `shared/db.py` | Supabase Postgres через `DATABASE_URL` |
| Дубли | PostgreSQL таблица `duplicates` | Supabase Postgres |
| Логи агентов | PostgreSQL таблица `agent_logs` | Supabase Postgres |
| Каналы и расходы | CSV | PostgreSQL/Supabase позже |
| Reels/источники/метрики | Пока план | PostgreSQL/Supabase позже |
| Транскрипты | Пока не храним | PostgreSQL/Supabase или Storage позже |
| Публикации | Redis-событие + Markdown | PostgreSQL/Supabase позже |

### Что добавить в будущую БД

| Таблица | Что хранит |
|---|---|
| `raw_leads` | сырые лиды до обработки |
| `leads` | квалифицированные лиды |
| `lead_scores` | скоринг и причины |
| `traffic_channels` | справочник каналов |
| `channel_costs` | расходы |
| `channel_facts` | лиды, встречи, сделки, выручка |
| `content_publications` | публикации Агента 4 |
| `content_performance` | просмотры, клики, лиды от контента |
| `source_items` | найденные источники: посты, объявления, Reels, тендеры |
| `reels_items` | метаданные Reels |
| `reels_analysis` | хуки, форматы, выводы |
| `transcripts` | расшифровки видео/аудио |
| `attribution_events` | связь источник -> лид -> сделка -> выручка |

### Как использовать Supabase

Supabase в этом проекте должен быть не второй отдельной базой, а облачным PostgreSQL через `DATABASE_URL`.

Простой принцип:

```text
SQLAlchemy + DATABASE_URL -> PostgreSQL
SQLAlchemy + Supabase DATABASE_URL -> Supabase Postgres
```

`SUPABASE_URL` и `SUPABASE_KEY` нужны только позже, если понадобятся Supabase API, Storage, Realtime или Auth.

## 7. Минимальный тест урока 4

Самый безопасный тест на ближайший этап:


Вариант A: лиды

```text
ручной RawLead fixture -> Agent 3 cleaner/enricher/scorer dry-run -> Agent 5 dry-run notification/report
```

Вариант B: источники

```text

```

Вариант C: контент


Рекомендация: начать с варианта A после `.env` и Redis, потому что проект всё ещё ждёт запуск MVP. Reels/контент можно делать вторым тестом.
