# Agent System Gap Check

Дата: 2026-05-06

Проект: `design-studio-lead-engine`

Статус: контрольная таблица. Ничего не удалялось, не переносилось, платные сервисы не запускались.

## 1. Эталонная схема из lessons

Для проекта лидогенерации в `lessons/system/agent-block-schemes-and-creation-map.md` выбрана схема:

```text
source-scout -> collector -> processor / enricher -> scorer -> CRM / router -> outreach / responder -> analytics / ROMI
```

Простыми словами:

1. Найти источники.
2. Собрать сырой лид или сигнал.
3. Очистить и обогатить.
4. Оценить важность.
5. Передать в CRM.
6. Довести до диалога или ответа.
7. Посчитать, что реально дало заявки, сделки и деньги.

## 2. Наша текущая система

В проекте есть 6 основных агентских отделов:

| Агент | Папка | Роль | Статус |
|---|---|---|---|
| Agent 1 Scout | `agents/agent1_scout/` | Разведка источников, конкурентов, рынка | Есть папки, runtime не подключён |
| Agent 2 Collector | `agents/agent2_collector/` | Сбор лидов и сигналов | Частично реализован |
| Agent 3 Processor | `agents/agent3_processor/` | Очистка, обогащение, скоринг, оффер | Реализован |
| Agent 4 Publisher | `agents/agent4_publisher/` | Контент и публикации | Частично реализован как внутренний контент-движок |
| Agent 5 CRM | `agents/agent5_crm/` | Bitrix24, уведомления, аналитика | Частично реализован |
| Agent 6 Outreach | `agents/agent6_outreach/` | Мониторинг, ответы, одобрение, отправка | Частично реализован |

## 5. Что уже является преимуществом

- Архитектура не хаотичная: есть 6 отделов, Redis-очереди, CRM, отчётность.
- Agent 3 уже разделён правильно: очистка, обогащение, скоринг, оффер.
- Agent 6 уже сделан осторожно: есть approver перед отправкой, sender с задержкой, lead detector.
- Agent 4 больше не отдельный проект `sales_ai`, а внутренний контент-движок.
- Есть сквозная аналитика каналов: CPL, CAC, profit, ROMI, окупаемость, рентабельность.
- Есть безопасный GitHub-подход: `.env` и сессии не должны уходить в репозиторий.

## 6. Главные разрывы

### Разрыв 1 — analytics / ROMI не закреплён как финальный агентский слой

Сейчас `analytics_reporter` уже считает каналы, но в архитектуре он пока выглядит как дополнительная функция Agent 5. По lesson-схеме это должен быть обязательный последний слой:

```text
источник -> лид -> CRM -> сделка -> выручка -> ROMI
```

Что нужно позже:

- связать события `content_published` с отчётом;
- сохранять не только канал, но и публикацию/идею контента;
- считать ROMI по каналам и по контенту.

### Разрыв 2 — Scout пока не передаёт задачи дальше

Agent 1 должен выдавать не просто “исследование”, а понятные входы:

- для Agent 2: где искать лиды;
- для Agent 4: какие офферы, темы, боли, форматы использовать;
- для Agent 5: какие источники добавить в аналитику.

Формат будущего выхода Scout:

```text
source_id, source_type, channel, query, why_interesting, owner_agent, next_action
```

### Разрыв 3 — много пустых collector/monitor папок

Это не ошибка, если мы идём по правилу “один источник сначала”. Но риск есть: пустые папки создают ощущение, что система готова шире, чем на самом деле.

Решение:

- не удалять;
- не запускать всё сразу;
- помечать статус `planned`, `manual`, `active`;
- сначала закрыть один источник.

### Разрыв 4 — Agent 4 может снова расползтись в отдельный проект

Исторически был `sales_ai`. Сейчас активного дубля нет, только архивы:

```text
../_archive_sales_ai_2026-05
../_archive_sales_ai_2026-05-final
```

Правило:

- не возвращать `sales_ai` как отдельный активный проект;
- полезные идеи переносить только внутрь Agent 4;
- не переносить курс, Jarvis, квизы, формы и лишнюю аналитику.

### Разрыв 5 — реальный запуск всё ещё блокируют доступы

По последним записям `REPORT.md`, базовый MVP всё ещё упирается в:

- `.env` обязательные поля;
- Redis;
- Telegram-сессии;
- Bitrix24.

Новые агенты не должны создаваться до закрытия этих блокеров, иначе будет больше хаоса.

## 7. Что лишнее или пока опасно

| Элемент | Почему не трогаем сейчас |
|---|---|
| Массовый Apify/Reels-сбор | Может стоить денег, нужен лимит и отдельное подтверждение |
| Реальные публикации во все каналы | Нет токенов и контроля дублей по всем площадкам |
| Supabase API слой | Пока достаточно `DATABASE_URL`; не нужна вторая логика хранения |
| Новые независимые проекты рядом | Риск дублей и потери контекста |
| Все collectors одновременно | Нарушает принцип “один источник сначала” |
| Video/audio/transcripts | Полезно позже, но сейчас отвлекает от MVP |

## 8. Рекомендуемый порядок

### Сейчас

Закрыть запуск MVP:

```text
.env -> Redis -> Telegram-сессии -> Bitrix24 -> один сквозной тест
```

### После первого сквозного теста

Усилить Agent 5 как финальный слой:

```text
analytics_reporter -> channels + content_published + deals + ROMI
```

### Потом

Подключать новые источники строго по одному:

```text
1 источник -> 3-5 тестовых записей -> проверка -> отчёт -> только потом следующий источник
```

## 9. Один маленький следующий шаг

Самый полезный следующий шаг:

```text
безопасно проверить .env по статусам SET/EMPTY и закрыть обязательные поля для MVP
```

Почему именно он:

- без `.env` и Redis не проверить реальную цепочку;
- без реального теста нельзя честно понять, какой агент ломается;
- новые документы и агенты уже достаточно описаны, теперь нужен первый рабочий проход.
