# Таблица маршрутизации обращений с сайта в Bitrix24 без формы

Дата: 2026-05-15.

Статус: рабочая таблица для настройки. Код сайта, Bitrix24, CRM, Open Lines, формы, внешние API, deploy и публикации не запускались.

## 0. Решения после уточнения

Уточнение от Яники от 2026-05-15:

- при первом обращении создаём не лид, а сделку;
- подключены почта, MAX, телефон, Telegram и открытые линии;
- сайт по-прежнему не использует форму сбора лидов.
- целевой ответственный по входящим сделкам — AI-менеджер первого касания, который ведёт диалог до результата или до передачи человеку.

Read-only проверка Bitrix24 от 2026-05-15 показала:

- webhook в локальном `.env` заполнен, значение не выводилось;
- справочник источников в Bitrix24 содержит `CALL`, `EMAIL`, `WEB`, `2|MAX`, `12|MAX_NUMBER_REDHAM`, `2|TELEGRAM`, `6|TELEGRAM_UNLIM` и другие источники;
- отдельных источников `site_phone`, `site_email`, `site_max`, `site_telegram`, `site_yandex_maps`, `site_2gis` в справочнике Bitrix24 не найдено;
- последние сделки распределены между ответственными Bitrix24 ID `1` и `8`;
- значит на MVP точный сайтовый источник нужно хранить в комментарии к сделке или в отдельном пользовательском поле, а стандартный `SOURCE_ID` Bitrix24 оставлять техническим каналом.

Что значит `ответственный`: это тот, кто обязан первым ответить клиенту и довести обращение до следующего шага.

Целевая схема:

```text
ответственный = AI-менеджер первого касания
контролёр = человек в Bitrix24, который принимает сложные случаи
```

Если в Bitrix24 пока нет отдельного пользователя `AI-менеджер`, временно сделка может назначаться на человека-контролёра, а в комментарии фиксируется: `первичный ответственный — AI-менеджер`.

Обновлено 2026-05-16: текущий CRM-webhook принадлежит пользователю `ID=1`. По решению Яники этот пользователь используется как администратор/человек-контролёр для первых AI-сделок, поэтому Agent 5 ставит:

```text
ASSIGNED_BY_ID=1
```

Подробное правило:

```text
docs/ai-deal-manager-responsibility.md
```

## 1. Нужна ли отдельная база данных

Для первого этапа отдельная база данных не нужна.

Главное хранилище обращений — Bitrix24.

Сайт не собирает заявки через форму. Он только даёт человеку понятные кнопки связи. Когда человек сам написал, позвонил или отправил письмо, обращение фиксируется в Bitrix24.

Отдельная база данных может понадобиться позже, только если будем делать автоматизацию Agent 5:

- хранить сырые входящие события до отправки в Bitrix24;
- хранить технические логи обработки;
- защищаться от дублей;
- строить отдельный отчёт по источникам;
- временно складывать обращения, если Bitrix24 недоступен.

На MVP это лишнее. Сначала достаточно Bitrix24 + ручной контроль первых обращений.

## 2. Главный принцип

Не ставим на сайт форму.

Правильный путь:

```text
кнопка на сайте
-> человек сам выбирает канал
-> пишет / звонит / открывает карту
-> менеджер или интеграция фиксирует обращение в Bitrix24
-> источник ставится по таблице ниже
```

### 4.1. Реальные пользовательские поля для Agent 5

Обновлено 2026-05-16: для сделок Bitrix24 созданы короткие поля `UF_CRM_VPP_*` через безопасный sync-скрипт `scripts/sync_bitrix_deal_custom_fields.py --execute`.

Важно: скрипт создал только поля в структуре CRM. Сделки, лиды, контакты, сообщения и рассылки не создавались.

Почему короткие: у пользовательских полей Bitrix24 есть ограничение на код поля, поэтому длинные коды вида `UF_CRM_FIRST_TOUCH_CHANNEL` не используем.

| Что хранит Agent 5 | Реальное поле Bitrix24 | Тип | Значения для API |
|---|---|---|---|
| Сайтовый источник | `UF_CRM_VPP_SITE_SRC` | строка | текст |
| Поток A/B | `UF_CRM_VPP_FLOW` | список | `A` -> `100`, `B` -> `102` |
| Скор лида | `UF_CRM_VPP_SCORE` | список | `hot` -> `104`, `warm` -> `106`, `cold` -> `108`, `off` -> `110` |
| Тип объекта | `UF_CRM_VPP_OBJ_TYPE` | строка | текст |
| Площадь | `UF_CRM_VPP_AREA_M2` | число | число в м2 |
| Первое касание | `UF_CRM_VPP_FIRST_CH` | строка | текст |
| Последнее касание | `UF_CRM_VPP_LAST_CH` | строка | текст |
| Страница входа | `UF_CRM_VPP_LAND_URL` | строка | путь страницы |
| Статус AI-менеджера | `UF_CRM_VPP_AI_STATUS` | строка | текст |

Для страховки те же данные всё равно можно кратко дублировать в `COMMENTS`, чтобы менеджеру было видно контекст без открытия всех полей.

Старые длинные логические названия `UF_CRM_SITE_SOURCE`, `UF_CRM_LEAD_FLOW`, `UF_CRM_FIRST_TOUCH_CHANNEL` и похожие не использовать как реальные поля Bitrix24. В коде Agent 5 использовать только `UF_CRM_VPP_*`.

### 4.2. Маршрут сделки по потоку

Agent 5 в MVP сам выбирает первую воронку и стадию по `flow`:

| Поток | Что это | `CATEGORY_ID` | `STAGE_ID` |
|---|---|---|---|
| `A` | перепланировки | `4` | `C4:NEW` |
| `B` | проектирование, ПД/РД, КР/КЖ/КМ, реконструкция | `0` | `NEW` |

Воронка `CATEGORY_ID=2` — `Исполнители` — не используется для входящих клиентских запросов с сайта. Её можно использовать позже для отдельного сценария поиска подрядчиков/исполнителей.

Ответственный для первых AI-сделок:

```text
ASSIGNED_BY_ID=1
```

Почему так: это владелец текущего CRM-webhook и администратор, которого Яника выбрала как принимающего первые заявки.

## 5. Как менеджеру фиксировать первые обращения

Для первых 10 обращений не автоматизировать вслепую.

Менеджеру достаточно задать 4 вопроса:

```text
1. Вы с сайта или из карт?
2. Какой город и какой объект?
3. Какая задача: проект, реконструкция, перепланировка, разделы, обследование?
4. Какие документы уже есть?
```

После этого в Bitrix24 создаётся или обновляется сделка, ставится `SOURCE_ID` из таблицы и короткий комментарий с сайтовой меткой.

## 6. Как не потерять источник без формы

Без формы источник можно сохранить тремя способами:

1. Ручной вопрос менеджера: `Вы с сайта?`.
2. Разные источники в Bitrix24 для разных каналов: `site_max`, `site_telegram`, `site_phone`.
3. Позже — отдельные безопасные ссылки или коды страниц, например `SITE-PD`, `SITE-RD`, `SITE-REK`.

В ссылках нельзя передавать персональные данные:

- телефон;
- email;
- ФИО;
- адрес объекта;
- кадастровый номер;
- документы;
- суммы.

## 7. Что делать с Bitrix24 дальше

Следующий безопасный порядок:

1. Создать или проверить справочник источников в Bitrix24.
2. Использовать сделку как основную карточку первого обращения.
3. Не создавать отдельные `site_*` источники вслепую: сначала хранить их в комментарии или пользовательском поле.
4. Проверить, какой из MAX/Telegram `SOURCE_ID` реально ставится автоматически из открытых линий.
5. Описать роль `AI-менеджер` и выбрать человека-контролёра.
6. Провести первые 10-20 обращений в режиме AI-черновика с ручным подтверждением.
7. После этого решить, какой один канал автоматизировать первым.

## 8. Что не делать сейчас

Не делать сейчас:

- форму на сайте;
- квиз;
- скрытую отправку в CRM;
- подключение Bitrix24 webhook без проверки прав;
- отдельную базу данных для MVP;
- автоматическое создание лидов из всех сообщений подряд;
- публикацию изменений на боевой сайт без отдельного подтверждения.

## 9. Следующий маленький шаг

Проверить в Bitrix24 перед первой реальной отправкой сделки:

```text
какой Bitrix24-пользователь будет техническим владельцем AI-сделок
кто будет человеком-контролёром
какой SOURCE_ID реально ставится у MAX из открытой линии
какой SOURCE_ID реально ставится у Telegram из открытой линии
```

Технический владелец для MVP уже выбран: `ASSIGNED_BY_ID=1`. Осталось проверить только реальные `SOURCE_ID` из открытых линий MAX/Telegram и затем сделать один реальный тест создания сделки `crm.deal.add` из локального тестового лида. До этого Agent 5 работает только в dry-run.
