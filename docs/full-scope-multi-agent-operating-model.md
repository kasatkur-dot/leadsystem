# Full-Scope Multi-Agent Operating Model

Дата: 2026-05-16

Статус: рабочая модель. Она не запускает рекламу, Bitrix24-запись, MAX, Telegram, VK, Яндекс Директ, тендерные площадки, Redis, scheduler, LLM, deploy или платные сервисы.

## 1. Главное решение

B2B-коммерция не заменяет весь проект.

Правильная модель такая:

```text
вся система ВПП
-> сохраняет все направления
-> внутри нее B2B-коммерция получает первый приоритетный спринт
-> остальные направления остаются в структуре и продолжают учитываться
```

Это значит:

- не создаем Agent 7;
- не копируем чужую схему `24 agents`;
- не ломаем текущие 6 агентов;
- не удаляем направления `дома`, `квартиры`, `перепланировки`, `реконструкции`;
- не переводим все только в коммерческие склады и ТЦ;
- добавляем внутри текущих агентов понятные сценарии, сегменты и артефакты.

## 2. Полный состав направлений

В проекте остаются все направления ВПП:

| Направление | Клиент | Поток | Статус в системе |
|---|---|---|---|
| Коммерческие объекты | юрлица, девелоперы, арендаторы, собственники бизнеса | B | первый приоритет роста |
| Склады / ангары / производство | юрлица | B | первый B2B-сегмент |
| ТЦ / магазины / fit-out | юрлица, арендаторы, сети | B + A | первый B2B-сегмент |
| Реконструкция | юрлица и частные собственники | B | высокий чек, ручная квалификация |
| Коммерческая перепланировка | юрлица | A + B | первый B2B-сегмент |
| Перепланировка квартир | физлица | A | сохраняем как стабильный поток |
| Частные дома | физлица | B | вторичный поток, не удаляем |
| Отдельные разделы / консультации | физлица и юрлица | A + B | только если профильный и выгодный запрос |

Поток `A` - перепланировки, согласование, безопасный путь по помещению.

Поток `B` - инженерное проектирование, ПД/РД, конструктив, КР/КЖ/КМ/АР, экспертиза при необходимости.

## 3. Как устроены агенты

Остаются 6 агентов. Внутри них могут быть роли и сценарии, но это не новые агенты.

| Агент | Роль | Что должен учитывать по полной системе |
|---|---|---|
| Agent 1 Scout | разведка рынка | конкуренты, цены, спрос, города, тендеры, B2B-сегменты, частные направления |
| Agent 2 Collector | сбор сырых лидов | тендеры, Avito, Profi, Яндекс Услуги, карты, email, старые заявки, площадки |
| Agent 3 Processor | очистка и квалификация | дубль или нет, поток A/B, объект, город, площадь, срочность, профильность |
| Agent 4 Publisher | контент и посадочные страницы | сайт, лендинги, MAX-канал, Telegram, VK, Дзен, карты, объявления, кейсы |
| Agent 5 CRM | Bitrix24 и контроль | сделки, источники, поля ВПП, статусы, SLA, отчеты, дубли, ROMI |
| Agent 6 Outreach | первое касание | Telegram/VK/MAX/TenChat/email/ABM только после approval и лимитов |

Пример простыми словами:

```text
Не делаем отдельного агента "Склады".
Делаем сценарий "Склады" внутри Agent 1, 2, 3, 4, 5 и 6.
```

## 4. Сценарии вместо новых агентов

Сценарий - это путь задачи через текущие 6 агентов.

| Сценарий | Что запускает | Какие агенты участвуют |
|---|---|---|
| B2B-заявка с лендинга | юрлицо пишет по складу, ТЦ, магазину, реконструкции | 4 -> 3 -> 5 |
| Тендер / коммерческий запрос | найдено письмо или площадка с проектированием | 2 -> 3 -> 5 |
| Перепланировка квартиры | человек спрашивает про план, БТИ, ремонт, согласование | 4/6/2 -> 3 -> 5 |
| Частный дом | запрос на проект дома, конструктив, разделы | 4/2/6 -> 3 -> 5 |
| Реконструкция | объект сложный, нужны документы и ручная оценка | 1/2/4 -> 3 -> 5 -> человек |
| Контент-доверие | пост, кейс, разбор ошибки, карточка компании | 1 -> 4 -> 5 analytics |
| CRM-гигиена | дубли, пустые источники, старые заявки | 5 |

## 5. Откуда берутся исходные данные

Исходные данные не должны жить "в голове". У каждого входа должен быть источник.

| Источник | Что приходит | Кто принимает |
|---|---|---|
| Основной сайт | обращения, переходы, страницы услуг, контакты | Agent 4 + Agent 5 |
| Отдельные лендинги | сегментные обращения по складам, ритейлу, реконструкции, перепланировке бизнеса | Agent 4 + Agent 5 |
| MAX-канал `СИЛА Проекта` | вопросы после постов, доверие, переходы | Agent 4 + Agent 5 |
| Telegram / VK | вопросы, комментарии, личные сообщения | Agent 4 / Agent 6 |
| Email | тендеры, ТЗ, документы, коммерческие запросы | Agent 2 |
| Bitrix24 | контакты, сделки, история, дубли, источники | Agent 5 |
| Avito / Profi / Яндекс Услуги | заявки и отклики | Agent 2 + Agent 4 |
| Яндекс Карты / 2GIS | звонки, переходы, отзывы, карточка компании | Agent 4 + Agent 5 |
| Коммерческие тендерные площадки | закупки, ТЗ, сроки, требования | Agent 1 + Agent 2 |
| ABM-аутрич | список компаний, точечное первое касание | Agent 1 + Agent 6 |
| Старые заявки и контакты | повторные сделки, дубли, база для очистки | Agent 5 |

## 6. Минимальные карточки данных

Чтобы система работала аккуратно, каждый этап должен оставлять понятный артефакт.

| Артефакт | Что это | Где используется |
|---|---|---|
| `source_card` | откуда пришел сигнал: сайт, MAX, email, тендер, карта | Agent 1/2/4/6 |
| `raw_lead` | сырой запрос до очистки | Agent 2/6 |
| `intake_card` | нормальная карточка первого обращения | AI-менеджер / Agent 3 |
| `qualified_lead` | очищенный и оцененный лид | Agent 3 |
| `bitrix_deal` | сделка в Bitrix24 | Agent 5 |
| `content_event` | пост, лендинг, карточка, объявление | Agent 4 |
| `crm_hygiene_report` | дубли, пустые поля, старые заявки | Agent 5 |
| `romi_report` | источник -> лид -> сделка -> деньги | Agent 5 |

## 7. Что значит `КП за 24 часа`

Это можно оставить как сильный оффер, но только с условием.

Правильная формулировка:

```text
КП за 24 часа при полном комплекте исходных данных.
```

Полный комплект для первичного КП:

- тип объекта;
- город;
- площадь;
- что нужно сделать;
- стадия: идея, куплено помещение, есть ТЗ, есть проект, нужна реконструкция;
- документы: план, БТИ, ТЗ, требования арендодателя, старый проект, фото, обмеры;
- срок;
- кто заказчик: физлицо, ИП, ООО, управляющая компания, девелопер;
- контакт для связи.

Если данных нет, AI-менеджер не обещает КП. Он собирает недостающие данные и передает человеку.

## 8. Как B2B-спринт встраивается в полную систему

B2B-спринт - это первый фокус на деньги и платежеспособных клиентов.

Он дает:

- отдельные лендинги под коммерческие сегменты;
- понятные офферы без опасных обещаний;
- поля в Bitrix24 для источника, потока, объекта, площади и статуса AI;
- контент под доверие и кейсы;
- будущий тест Яндекс/VK/тендеров только после отдельного решения.

Но он не удаляет:

- перепланировки квартир;
- частные дома;
- реконструкции;
- старые заявки;
- карты;
- MAX-канал;
- контентную воронку;
- Bitrix24-гигиену.

## 9. Что нужно перестроить

Перестроить нужно не агентов, а слой управления направлениями.

Добавить в систему:

| Что добавить | Зачем |
|---|---|
| `segment` | склад, ритейл, реконструкция, квартира, дом, другое |
| `client_type` | физлицо, ИП, ООО, УК, девелопер, арендатор |
| `flow` | A или B |
| `source` | MAX, сайт, email, тендер, карта, Avito, Profi, old_base |
| `landing_id` | с какой посадочной страницы пришел человек |
| `data_completeness` | хватает ли данных для КП за 24 часа |
| `next_action` | что делать дальше: запросить данные, позвонить, КП, отказ, архив |

Эти поля должны идти через Agent 3 и Agent 5 в Bitrix24/dashboard.

## 10. Что оставить как было

Оставляем:

- 6 агентов;
- два основных потока `A` и `B`;
- Agent 5 как главный слой Bitrix24;
- Agent 4 как контент, лендинги и точки доверия;
- ручной approval перед публикациями и аутричем;
- запрет на обязательные формы/квизы/лид-магниты в обмен на контакт;
- запрет на реальную рекламу и платные сервисы без отдельного подтверждения;
- правило: сначала dry-run, потом один ручной тест, потом масштабирование.
