# Three Qualified Leads Sprint From Astrology Method

Дата: 2026-05-16

Цель: получить первые 3 качественные заявки на проектирование, не обещая гарантированный результат и не используя опасный outreach.

Источник методики:

- `docs/lecture-2026-05-15-growth-and-agent-system-for-vpp.md`
- блок про AI-астрологию: эксперт + продукт + контент-завод + реклама/источники + регистрация + прогрев + деньги;
- блок про 8000 подписчиков: много источников + агенты + 3-4 прогревающих касания в день + аналитика.

## 1. Как именно делали канал астрологии

Не тема важна, а схема:

```text
эксперт с опытом
-> цифровой продукт / приложение
-> короткий контент
-> просмотры и подписчики
-> сайт / регистрация
-> прогрев
-> платная заявка / покупка
```

У ВПП аналог:

```text
проектная организация с СРО и 10+ лет опыта
-> AI-проверка исходных данных к КП
-> короткий экспертный контент
-> MAX / будущий лендинг / тендерный вход
-> Bitrix24
-> КП / список недостающих данных
-> договор
```

## 2. Что значит “качественная заявка”

Заявка считается качественной, если есть минимум:

- тип объекта;
- город;
- площадь или примерный масштаб;
- кто клиент: юрлицо, ИП, физлицо;
- что нужно: ПД, РД, перепланировка, реконструкция, обследование, отдельный раздел;
- срок;
- контакт;
- есть или нет исходные данные.

Не считаем качественной заявкой:

- “сколько стоит проект?” без объекта;
- пустой контакт без задачи;
- неподходящий запрос не по проектированию;
- человек без бюджета и без объекта.

## 3. 3 заявки: рабочая воронка

```text
12-15 единиц контента
-> 3 сегментных входа
-> 3 источника
-> 3 сценария AI-менеджера
-> 3 Bitrix-preview
-> 3 ручных решения: КП / уточнить / отказ
```

Первые 3 сегмента:

1. Коммерческая перепланировка.
2. Склады / ангары.
3. Реконструкция.

## 4. 3-недельная логика

### Неделя 1. Упаковка и прогрев

- довести 5 MAX B2B-постов;
- подготовить 3 коротких экспертных разбора;
- подготовить структуру 3 лендингов;
- проверить Bitrix24-поля и источники;
- не запускать рекламу.

Цель недели: получить первые вопросы или готовые материалы для публикации.

### Неделя 2. Источники и ручные тесты

- один approved MAX-пост;
- один read-only тендерный мониторинг;
- один список ABM-компаний без рассылки;
- один сценарий AI-менеджера на каждый сегмент.

Цель недели: получить 1-2 входящих вопроса или конкретные кандидаты на заявку.

### Неделя 3. Усиление

- повторить лучший угол;
- подготовить второй пост по теме, где была реакция;
- вручную обработать входящие вопросы;
- считать `source_data_received`, `kp_ready`, `kp_sent`.

Цель недели: довести до 3 качественных заявок или понять, какой сегмент не работает.

## 5. Роли специалистов и агентов

Новых агентов не создаём. Используем 6 текущих агентов как роли команды.

| Роль как у лектора | У нас | Что делает |
|---|---|---|
| стратег / предприниматель | оркестратор + Яника | выбирает сегмент и разрешает внешние действия |
| маркетолог | Agent 4 + Agent 5 | контент, UTM, метрики |
| OSINT / разведка | Agent 1 | ищет источники и площадки |
| сборщик заявок | Agent 2 | принимает сигналы и raw lead |
| квалификатор | Agent 3 | проверяет исходные данные и готовность к КП |
| CRM / аналитик | Agent 5 | Bitrix24, дубли, ROMI |
| outreach-специалист | Agent 6 | готовит сообщения, но не отправляет без approval |

## 6. Что должен делать каждый агент в этом спринте

### Agent 1 Scout

- найти 10 конкретных площадок/источников для B2B;
- разделить их на `safe`, `needs_approval`, `do_not_use`;
- не писать людям.

### Agent 2 Collector

- подготовить структуру входящего raw lead;
- проверить, какие поля нужны от источника;
- не собирать массово.

### Agent 3 Processor

- для каждого сегмента дать список недостающих данных;
- определить `kp_ready=true/false`;
- подготовить первый ответ AI-менеджера.

### Agent 4 Publisher

- довести 5 MAX-постов;
- подготовить 3 коротких разбора;
- ничего не публиковать без approval.

### Agent 5 CRM

- держать Bitrix24-метки;
- считать заявки и качество;
- не менять карточки без approval.

### Agent 6 Outreach

- подготовить approved-only сообщения;
- не отправлять;
- дать стоп-правила.

## 8. Что будет результатом системы

Через dashboard должно быть видно:

- какие сегменты выбраны;
- какие материалы готовы;
- какие источники готовы;
- что можно делать только после approval;
- сколько заявок пришло;
- почему заявка качественная или нет;
- какой следующий шаг.

## 9. Что нельзя забыть

- Астрокейс работал потому, что был эксперт и продуктовый вход. У ВПП экспертность есть, продуктовый вход должен быть “проверка исходных данных к КП”.
- Не гнаться за подписчиками ради подписчиков. Нужны заявки с объектом и исходными данными.
- Не запускать весь интернет сразу. Сначала 3 сегмента и 3 источника.
- Не создавать Agent 7. Усиливать текущих 6 агентов сценариями.
