# ТЗ/промпт для Claude Design: новый сайт Вектор Плюс-Про

Дата создания: 2026-05-07.

Статус: внутренний документ и источник требований. Первая локальная версия сайта уже создана в `site/` и редактируется. Vercel не запускался, сайт не опубликован.

## 1. Зачем нужен этот документ

Этот документ нужен для следующего шага по сценарию lessons:

```text
исследование -> структура сайта -> ТЗ для дизайна -> Claude Design -> код -> локальная проверка -> GitHub -> Vercel preview -> production после подтверждения
```

Простыми словами: сначала мы даём нейро-дизайнеру очень понятное задание, чтобы он не фантазировал, не делал шаблонный сайт и не придумывал неподтверждённые услуги.

## 2. Что мы передаём в Claude Design

Передаём не весь проект и не секреты, а только безопасный публичный контекст:

- название компании;
- ИНН и ОГРН;
- проверенные публичные ссылки;
- структуру сайта;
- ограничения по формулировкам;
- требования к визуалу;
- список страниц первой волны;
- требования к мобильной версии.

Нельзя передавать:

- `.env`;
- токены;
- пароли;
- закрытые клиентские данные;
- договоры;
- закрытые чертежи;
- внутренние логи;
- Redis/Bitrix24/webhook-данные.

## 4. Главная цель сайта

Сайт должен отвечать на 4 вопроса:

1. Кто такая компания `Вектор Плюс-Про`.
2. Какие проектные задачи она решает.
3. Почему ей можно доверять.
4. Как обратиться без лишнего сложного квиза.

Сайт должен работать сразу для трёх аудиторий:

- человек-клиент;
- поисковые системы;
- AI-ассистенты и нейропоиск.

Главная GEO/AI-задача:

```text
ООО Вектор Плюс-Про -> ИНН/ОГРН -> новый сайт -> СРО -> услуги -> кейсы -> СИЛА Проекта -> Яндекс Карты / 2GIS
```

## 5. Аудитории сайта

Основные аудитории:

- собственники квартир и домов;
- предприниматели и владельцы коммерческих помещений;
- застройщики;
- генподрядчики;
- бюджетные организации;
- люди, которым нужно понять, какие документы нужны до ремонта, реконструкции или строительства.

Что важно аудитории:

- понятно, можно ли нам доверять;
- есть ли проектная компетенция;
- есть ли СРО;
- какие документы они получат;
- что можно сделать законно;
- что зависит от исходных данных;
- как быстро получить первый ответ.

## 6. Визуальное направление

Нужен не шаблонный “ремонтный” сайт и не легкомысленный лендинг.

Стиль:

- инженерный;
- спокойный;
- уверенный;
- современный;
- премиальный без пафоса;
- понятный человеку без технического образования;
- с ощущением порядка, точности и надёжности.

Визуальные ассоциации:

- чертёж;
- сетка;
- архитектурная линия;
- конструктив;
- документы;
- контроль;
- прозрачный процесс.

Чего избегать:

- фиолетового AI-шаблона;
- случайных 3D-иконок;
- стоковых людей в касках;
- чрезмерного “люкса”;
- агрессивных продаж;
- обещаний “100% согласуем”.

Желательное ощущение:

```text
Это не просто сайт услуг. Это аккуратная проектная система, где клиент понимает путь от вопроса до документации.
```

## 7. Цвета, типографика и графика

Рекомендация по цветам:

- основной тёмный: графит / почти чёрный;
- фоновый: тёплый светлый серый или молочный;
- акцент: спокойный инженерный синий, стальной, зелёно-серый или терракотовый;
- дополнительный: тонкие линии, сетка, карточки, полупрозрачные слои.

Типографика:

- заголовки крупные, спокойные, уверенные;
- текст читабельный;
- не использовать слишком много размеров;
- максимум 3-4 уровня текста;
- цифры и реквизиты должны выглядеть как важные доказательства, а не как мелкая сноска.

Графика:

- линии чертежей;
- сетки;
- схемы этапов;
- карточки документов;
- условные планы/узлы без реальных клиентских чертежей;
- аккуратные пиктограммы разделов `ПД`, `РД`, `КР`, `КЖ`, `КМ`.

## 8. Первая волна страниц

Нужно спроектировать сайт так, чтобы первая волна включала:

| Страница | Назначение |
|---|---|
| Главная | главный вход и доверие |
| О компании | юридическая и смысловая связка бренда |
| СРО и документы | доказательства компетенции |
| Услуги | обзор направлений |
| Проектная документация | страница под `ПД` |
| Рабочая документация | страница под `РД` |
| КР, КЖ, КМ | единая осторожная страница по конструктиву |
| Проектирование перепланировок | B2C/B2B-запросы по перепланировкам |
| Реконструкция | B2B/бюджетный поток |
| Кейсы | доказательства опыта |
| Контакты | понятный способ обратиться |

В первой версии не дробить `КЖ` и `КМ` на отдельные страницы, пока не подтверждены публичные кейсы и формулировки.

## 9. Структура главной страницы

Главная страница должна состоять из блоков:

1. Первый экран.
2. Кому помогаем.
3. Основные услуги.
4. Доказательства доверия.
5. Как проходит работа.
6. Кейсы.
7. Что важно знать до начала проекта.
8. Канал `СИЛА Проекта`.
9. Контакты.

```text
Вектор Плюс-Про — студия проектирования.
Проектируем архитектуру и конструкции, готовим проектную и рабочую документацию, помогаем с проектными решениями для перепланировок, реконструкций, коммерческих и частных объектов.
```

Кнопки:

- `Обсудить объект`;
- `Посмотреть услуги`;
- `Смотреть кейсы`.

Не делать навязчивую форму в первом экране.

Карточки:

- собственникам;
- предпринимателям;
- застройщикам;
- генподрядчикам;
- бюджетным организациям.

### 9.3. Основные услуги

Карточки:

- Проектная документация;
- Рабочая документация;
- КР, КЖ, КМ;
- Перепланировки;
- Реконструкция;

### 9.4. Доказательства доверия

Разместить:

- ИНН;
- ОГРН;
- СРО;
- Яндекс Карты;

- канал `СИЛА Проекта`;

### 9.5. Как проходит работа

Схема:

```text
заявка -> исходные данные -> анализ объекта -> состав работ -> договор -> проектирование -> проверка -> выдача документации -> сопровождение
```

### 9.6. Кейсы

Пока можно использовать шаблонные карточки-заготовки без реальных адресов:

- регион;
- стадия;
- разделы;
- задача;
- результат.

Нельзя придумывать реальные кейсы, адреса, площади и цифры.

### 9.7. Канал СИЛА Проекта

Блок:

```text
СИЛА Проекта — экспертный канал команды Вектор Плюс-Про.
Простым языком разбираем перепланировки, оформление домов, проектную и рабочую документацию, реконструкции и экспертизу.
```

## 10. Требования к страницам услуг

Каждая страница услуги должна иметь одинаковую понятную логику:

1. Что это такое.
2. Кому нужно.
3. Какие задачи решает.
4. Что входит в результат.
5. Какие исходные данные нужны.
6. Какие ограничения есть.
7. Какие документы получает клиент.
8. Связанные кейсы.
9. CTA на консультацию.

Тон текста:

- просто;
- честно;
- без “канцелярита”;
- без обещаний, которые зависят от госорганов, банка или состояния объекта.

## 11. Блоки, которые нельзя делать

Не делать:

- квиз с обязательным телефоном;
- лид-магнит “получи чек-лист только за номер”;
- автогарантии согласования;
- таймеры срочности;
- “оставьте заявку за 30 секунд, иначе цена вырастет”;
- фейковые отзывы;
- выдуманные кейсы;
- стоковые чертежи, похожие на реальные документы клиентов;
- блоки с неподтвержденными цифрами.

## 12. Мобильная версия

Мобильная версия обязательна.

На телефоне должно быть удобно:

- понять, кто компания;
- увидеть услуги;
- увидеть ИНН/ОГРН/СРО;
- открыть карты;
- перейти в канал;
- нажать телефон/email/MAX-аккаунт/Telegram;
- прочитать этапы работы;
- посмотреть кейсы.

Не делать на мобильной версии:

- длинные горизонтальные таблицы без адаптации;
- мелкий текст;
- огромные тяжёлые анимации;
- формы на 20 полей.

## 13. SEO/GEO/AI-требования к дизайну

Дизайн должен предусмотреть, что важные данные будут обычным текстом, а не только картинками.

Обязательно текстом:

- `ООО «Вектор Плюс-Про»`;
- `ИНН 0100005517`;
- `ОГРН 1230100001812`;
- услуги;
- СРО;
- адрес и контакты;
- Яндекс Карты;
- 2GIS;
- канал `СИЛА Проекта`.

Нужны места под:

- H1;
- H2;
- хлебные крошки;
- внутренние ссылки между услугами;
- FAQ на страницах услуг;
- блоки кейсов;
- блок “связанные услуги”.

## 14. Что Claude Design должен выдать

Идеальный результат от Claude Design:

- дизайн главной страницы;
- шаблон страницы услуги;
- шаблон страницы кейса;
- шаблон страницы `О компании`;
- шаблон страницы `Контакты`;
- адаптивная версия;
- понятная структура компонентов;
- чистый код без секретов;
- возможность потом перенести в `web/` или `site/` внутри проекта.

Если Claude Design может выдать код, пусть делает статический фронтенд без внешних платных API.

Не подключать:

- CRM;
- Bitrix24;
- Supabase;
- аналитику;
- формы с отправкой;
- платные API;
- трекеры без согласования.

Формы можно сделать только как визуальные блоки-заготовки без реальной отправки.

## 16. Многоагентный handoff сайта

Этот сайт делается по многоагентной логике, но без создания новых агентов.

Простая схема:

```text
Agent 1 Scout / Радар
-> Agent 4 Publisher / Креатор
-> Claude Design / Визуализатор
-> Codex / Сборщик
-> Человек / Главный редактор
-> Agent 5 CRM/Analytics / ROMI-аналитик позже
```

Рабочие имена нужны только для понятности. Они не создают новых агентов в папке `agents/`.

| Рабочее имя | Кто это | За что отвечает |
|---|---|---|
| `Радар` | Agent 1 Scout | Видит рынок, конкурентов, запросы, референсы, AI-ответы |
| `Креатор` | Agent 4 Publisher / creator | Упаковывает сайт, ТЗ, тексты, промпты, промо-идеи |
| `Визуализатор` | Claude Design | Делает визуальный макет сайта и, если возможно, фронтенд-код |
| `Медиа-цех` | HyperFrames / Remotion / HeyGen позже | Делает видео, анимации, озвучку и ролики, но не в первой волне |
| `Сборщик` | Codex | Вставляет код локально, проверяет, чинит, готовит к preview |
| `Главный редактор` | Яника / человек | Утверждает смысл, факты, стиль, публикацию |
| `ROMI-аналитик` | Agent 5 CRM/Analytics | Позже считает заявки, сделки, выручку, CPL, CAC, ROMI |

### 16.1. Agent 1 Scout

Agent 1 отвечает за разведку и факты.

Что он даёт для сайта:

- какие запросы ищут клиенты;
- каких конкурентов и AI-ответы нужно учитывать;
- какие услуги важны для GEO/AI-видимости;
- какие публичные доказательства нужны: СРО, Яндекс Карты, 2GIS, кейсы, внешние источники;
- какие формулировки нельзя писать без подтверждения.

Выход Agent 1:

```text
проверенные источники -> выводы -> требования к сайту
```

### 16.2. Agent 4 Publisher / creator

Agent 4 отвечает за упаковку, тексты и передачу задачи дизайнеру.

`creator` — это не новый агент, а режим работы внутри Agent 4.

Что делает Agent 4:

- собирает source-of-truth;
- формирует структуру сайта;
- готовит ТЗ на экраны;
- готовит промпт для Claude Design;
- следит, чтобы сайт не был шаблонным и не обещал лишнего;
- после дизайна помогает привести тексты, CTA и блоки доверия к единому стилю.

Выход Agent 4:

```text
структура сайта -> дизайн-brief -> промпт для Claude Design
```

### 16.3. Что получает Claude Design

Claude Design получает только безопасный публичный пакет:

- контекст компании;
- структуру страниц;
- требования к стилю;
- ограничения по текстам;
- список первой волны страниц;
- требования к мобильной версии;
- публичные ссылки: Яндекс Карты, 2GIS, СРО, канал.

Claude Design не получает:

- `.env`;
- токены;
- клиентские документы;
- закрытые чертежи;
- CRM/Bitrix24/Supabase-доступы;
- внутренние логи.

Ожидаемый выход Claude Design:

```text
дизайн-концепция -> страницы/шаблоны -> адаптивная версия -> код, если инструмент умеет
```

### 16.4. Что делает Codex после получения кода

Codex не публикует код сразу.

Codex должен:

- сохранить код в отдельную папку будущего сайта;
- проверить, нет ли секретов и клиентских данных;
- проверить структуру файлов;
- запустить локально, если это безопасно;
- проверить мобильную версию;
- проверить тексты, ссылки, ИНН, ОГРН, СРО, Яндекс Карты и 2GIS;
- убрать неподтверждённые обещания;
- подготовить проект к GitHub/Vercel preview только после отдельного подтверждения.

Выход Codex:

```text
проверенный локальный сайт -> список замечаний -> готовность к preview
```

### 16.5. Что проверяет человек перед публикацией

Перед публикацией человек проверяет смысл и юридическую аккуратность.

Нужно подтвердить:

- можно ли писать `официальный канал ООО` про `СИЛА Проекта`;
- актуальность СРО;
- какие услуги реально оказываются;
- можно ли прямо указывать `КР`, `КЖ`, `КМ`;
- можно ли писать про авторский надзор;
- можно ли писать про сопровождение экспертизы;
- какие кейсы можно публиковать;
- какие контакты публичные;
- что нет обещаний “100% согласуем”.

Без ручного подтверждения:

```text
production-публикация запрещена
```

### 16.6. Что позже учитывает Agent 5 CRM/Analytics

Agent 5 подключается позже, когда сайт уже прошёл первый безопасный этап.

Что должен учитывать Agent 5:

- источник заявки;

- UTM-метки;
- канал: сайт, Яндекс Карты, 2GIS, канал, AI-referrer позже;
- тип услуги;
- статус лида;
- сделку;
- выручку;
- расходы;
- CPL, CAC, ROMI.

В первой версии сайта Agent 5 не подключаем к реальной форме.

Сначала достаточно предусмотреть будущие поля:

```text
source
landing_page
utm_source
utm_medium
utm_campaign
service_interest
lead_status
deal_id
revenue
```

## 17. Медиа-упаковка сайта

Медиа-упаковка нужна, чтобы сайт потом можно было превратить не только в страницы, но и в понятные короткие материалы для канала `СИЛА Проекта`, MAX, Telegram, VK, Дзена и рекламных/социальных превью.

Важно: на этом этапе мы только закладываем места и требования. Видео, озвучку, платные инструменты и публикации не запускаем.

Короткий чек-лист для дизайна:

| Что учесть | Что должно быть в макете | Статус для первой волны |
|---|---|---|
| Видео/анимация | место под лёгкую анимацию первого экрана или отдельный промо-блок | заложить место, не делать тяжёлое видео |
| Путь клиента | визуальная схема от вопроса до выдачи документации | сделать сразу |
| Звук | только ручной запуск в будущем промо-видео | автозвук запрещён |
| Короткие ролики | логика нарезки сайта на MAX/Telegram/VK/Дзен | заложить темы и форматы |
| Social preview | обложки главной, услуги и кейса | заложить шаблоны |
| Инструменты | HyperFrames, Remotion, HeyGen/аналоги, ffmpeg, Whisper | оставить на потом |
| Ответственные | Agent 1, Agent 4, Claude Design, Codex, человек, Agent 5 | зафиксировать роли |

Главное правило:

```text
сайт сначала должен быстро и понятно продавать доверие, компетенцию и первый контакт;
медиа-слой усиливает это позже, но не должен замедлить первую версию.
```

### 17.1. Видео или анимация для первого экрана

В дизайне нужно предусмотреть один из двух безопасных вариантов:

- лёгкая анимация на первом экране: чертёжные линии, сетка, карточки документов, путь от объекта к документации;
- отдельный промо-блок ниже первого экрана: короткое видео-заглушка или место под будущий ролик.

В первой волне лучше выбрать лёгкую визуальную анимацию или статичный промо-блок. Тяжёлое видео на главном экране пока не нужно, чтобы сайт быстро открывался на телефоне.

### 17.2. Визуализация пути клиента

На сайте обязательно нужна понятная схема:

```text
вопрос клиента
-> исходные данные
-> анализ объекта
-> состав работ
-> договор
-> проектирование
-> проверка
-> выдача документации
-> сопровождение
```

Эта схема должна быть не просто текстом, а визуальным блоком: карточки, линия процесса, чертёжная сетка или “маршрут” по этапам.

### 17.3. Звук

На сайте нельзя включать звук автоматически.

Правило:

```text
звук только в отдельном промо-видео, которое человек запускает сам
```

Автозвук на сайте не использовать, потому что он раздражает пользователя, ухудшает доверие и может мешать мобильной версии.

### 17.4. Короткие ролики для каналов

После готового дизайна сайта можно будет сделать короткие ролики:

- MAX: экспертные короткие ролики и анонсы кейсов;
- Telegram: короткие видео/кружки/вертикальные ролики с объяснением услуги;
- VK: вертикальные клипы и обложки постов;
- Дзен: ролики и карточки “разбор ситуации простыми словами”.

Темы роликов должны выходить из страниц сайта:

- что такое ПД;
- чем РД отличается от ПД;
- зачем нужны КР, КЖ, КМ;
- как проходит проект перепланировки;
- почему нельзя обещать “100% согласуем”;
- какие исходные данные нужны до проекта.

### 17.5. Социальные превью сайта

Для сайта нужно предусмотреть будущие social preview:

- обложка главной страницы;
- обложка страницы услуги;
- обложка кейса;
- картинка для публикации ссылки в MAX/Telegram/VK;
- картинка для Дзена.

Превью должны выглядеть как часть единой системы: тот же стиль, цвета, линии, сетка, аккуратные инженерные карточки.

### 17.6. Инструменты на потом

Инструменты не запускаем сейчас, но оставляем как будущий слой:

- `HyperFrames` — для HTML-видео, промо-роликов, титров, субтитров, визуализации пути клиента;
- `Remotion` — для программных видео на React, если сайт будет на React/Next и нужны повторяемые шаблоны роликов;
- `HeyGen` или аналоги — для видео с ведущим/аватаром только после отдельного решения и проверки бюджета;
- `ffmpeg` — для технической сборки/нарезки видео;
- `Whisper` — для расшифровки будущих видео/аудио и подготовки субтитров;
- `Browser Use` — для безопасной визуальной QA-проверки сайта и preview, когда сайт будет локально запущен.

Платные инструменты, генерация голоса, аватары, автопубликация и реальные рекламные ролики запускаются только после отдельного подтверждения.

### 17.7. Первая волна и потом

В первой волне делаем:

- место под промо-блок или лёгкую анимацию;
- визуальную схему пути клиента;
- требования к social preview;
- запрет автозвука;
- возможность позже вынуть из сайта короткие ролики.

На потом оставляем:

- полноценное промо-видео;
- озвучку;
- ролики для MAX/Telegram/VK/Дзена;
- шаблоны видео в HyperFrames/Remotion;
- HeyGen/аватар;
- ROMI-аналитику по видео и контенту.

### 17.8. Кто за что отвечает

`Радар` / Agent 1 Scout:

- ищет примеры конкурентов, AI-ответы и визуальные паттерны рынка;
- собирает темы, из которых можно делать ролики и промо-блоки;
- проверяет, какие обещания нельзя писать без доказательств.

`Креатор` / Agent 4 Publisher:

- превращает выводы Agent 1 в медиа-ТЗ;
- пишет сценарии промо-блоков, коротких роликов, обложек и CTA;
- следит, чтобы стиль сайта, канала и роликов был единым.

`Визуализатор` / Claude Design:

- закладывает место под промо-блок, визуализацию пути клиента и social preview;
- не подключает реальные видео, звук, API и автопубликацию.

`Медиа-цех` / HyperFrames, Remotion, HeyGen позже:

- делает промо-видео, анимации, озвучку и ролики только после отдельного подтверждения;
- не запускает платные генерации и не публикует материалы без разрешения.

`Сборщик` / Codex:

- после получения кода проверяет, что нет автозвука, тяжёлых лишних видео, секретов и клиентских данных;
- проверяет мобильность, скорость, доступность и безопасность;
- не публикует сайт без ручного подтверждения.

`Главный редактор` / Яника:

- утверждает визуальный стиль, тексты, голос бренда, кейсы и право использовать формулировки.

`ROMI-аналитик` / Agent 5 CRM/Analytics позже:

- считает, какие страницы, ролики и каналы приводят заявки;
- связывает `source -> landing_page -> lead -> deal -> revenue -> ROMI`;
- отдельно учитывает эффективность коротких роликов и промо-видео.
