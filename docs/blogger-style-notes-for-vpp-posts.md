# Blogger Style Notes For VPP Posts

Дата: 2026-05-16

Статус: локальные правила стиля для постов. Не публикация.

## Что берём у блогеров и конкурентов

Хороший пост по проектированию и перепланировке обычно не начинается с услуги.

Он начинается с ситуации:

```text
аренда почти подписана
ремонт уже завтра
банк попросил документы
УК попросила проект
подрядчик ждёт решение
```

Потом показывает риск:

```text
красивый план не равен проекту
цена по площади может быть ошибкой
без исходных данных КП будет гаданием
требования арендодателя могут остановить ремонт
```

Потом даёт короткий список:

```text
проверьте 5 пунктов
соберите 7 исходных данных
задайте 3 вопроса до ремонта
```

И только потом мягкий переход:

```text
можно начать с проверки исходных данных
```

## Формула поста ВПП как один из вариантов

```text
живая ситуация
-> где обычно ломается процесс
-> простое объяснение
-> 3-5 проверок
-> честное ограничение
-> мягкий CTA
```

Это не единственный сценарий.

Если использовать его каждый раз, канал станет одинаковым и скучным.

## Правило разных концовок

Формат должен отличаться не только первым абзацем.

У каждого формата должна быть своя механика:

- у `разбора живой фразы клиента` можно вести к вопросу или отправке данных;
- у `вопрос-ответ` можно закончить коротким уточнением;
- у `анти-чеклиста` можно закончить самопроверкой;
- у `дневника проектировщика` лучше закончить наблюдением, заметкой дня, сохранением поста или вопросом к читателю;
- у `красного флага` финал должен быть коротким и резким;
- у `мини-кейса` финал должен опираться на вывод из ситуации.

Запрещено: менять только начало поста, а в конце каждый раз писать одну и ту же связку `почему это важно -> КП -> пришлите план`.

Agent 4 должен чередовать форматы:

```text
основной формат: разбор живой фразы клиента
вопрос-ответ
анти-чеклист
дневник проектировщика
мини-кейс без персональных данных
один термин простым языком
два пути
красный флаг
пост-карусель / список кадров
позиция эксперта
```

## Как это должен применять Agent 4

Agent 4 не должен просто писать полезный пост по теме.

Agent 4 не должен писать все посты по одной схеме.

Он должен выбрать конкретный рабочий контекст:

```text
собственник помещения
арендатор в ТЦ
директор склада
застройщик
ГИП или подрядчик
частный клиент с перепланировкой
```

И связать пост с одним следующим шагом:

```text
прислать план
описать объект
собрать исходные данные
задать вопрос
начать с первичной проверки
сохранить пост
переслать партнёру
попросить список документов
перейти к КП после исходных данных
```

Если тема слишком широкая, Agent 4 обязан сузить её до одного сегмента и одной ситуации.

Если предыдущий пост был ситуационным, следующий лучше делать в другом формате: чеклист, диалог, миф, ошибка, сравнение или термин.

Любой формат должен вести к монетизации прямо или косвенно:

```text
входящий лид
квалифицированная заявка
КП
сделка
доверие к каналу
сохранение
репост
партнёрская рекомендация
сигнал спроса для будущего оффера
```

Следующий шаг должен соответствовать формату.

Например, в `дневнике проектировщика` не надо каждый раз вести к КП. Там допустимы сохранение поста, вопрос в комментарии, заметка для собственника или обещание следующего разбора.

## Чего избегать

- Не начинать с “мы оказываем услуги”.
- Не писать сухо как КП.
- Не обещать согласование.
- Не писать “дешево и быстро”.
- Не пугать без пользы.
- Не делать обязательный лид-магнит.
- Не писать “мы всегда готовы помочь”.
- Не писать “не стоит рисковать” без конкретной проверки.
- Не писать “получить консультацию” как финальный CTA.
- Не писать “без лишних проблем” и “избежать неприятных сюрпризов”.
- Не обещать цену, срок, КП или экспертизу без исходных данных.
- Не использовать неподтвержденные кейсы и цифры.
- Не использовать “СНиПы” как общее слово для всех норм.
- Не использовать длинное тире. Если фраза требует паузы, разбить её на два предложения.

## B2B-сегменты для первых волн

```text
commercial_replan: магазины, офисы, салоны, помещения в ТЦ, требования УК/арендодателя
warehouse_design: склады, ангары, производство, логистика, нагрузки, сети, технология
reconstruction: школы, детские сады, МКД, общественные и коммерческие здания
pd_rd: проектная документация, рабочая документация, экспертиза, стройка
kr_kzh_km: конструктив, железобетон, металлоконструкции, отдельные разделы
private_secondary: частные дома и квартирные перепланировки как дополнительный поток
```

## Проверка качества черновика

Перед approval текст должен отвечать на 5 вопросов:

```text
Кто герой ситуации?
Что у него сломалось или может сломаться?
Соответствует ли структура выбранному формату?
Не повторяет ли конец прошлые посты?
Какой один следующий шаг подходит именно этому формату?
```

## Тон

- экспертно;
- спокойно;
- с конкретикой;
- без канцелярита;
- как опытный практик объясняет собственнику или директору.

## Пример удачного начала

```text
Представьте: помещение под магазин уже нашли.
Аренда почти согласована, дизайнер нарисовал план, подрядчик готов выходить на объект.
И тут управляющая компания говорит: принесите проект.
```
