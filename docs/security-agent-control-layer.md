# Security Agent Control Layer

Источник правил: `agents/Агент безопастности/AGENT_SECURITY.md`.

Это не Agent 7. Это контрольный слой безопасности, который применяется ко всем 6 агентским отделам, checker/dashboard, LLM, MCP/API и будущим интеграциям.

## Зачем нужен

Агент безопасности отвечает не за лиды и не за продажи, а за границы: что можно делать автоматически, что можно делать только локально, а что требует отдельного подтверждения Яники.

Он закрывает риски:

- утечка токенов, cookies, session-файлов, webhook URL и `.env`;
- случайная отправка данных клиентов во внешние LLM/API;
- платные запуски без подтверждения;
- реальные публикации, outreach и CRM-действия без approval;
- prompt injection, lethal trifecta, SSRF, slopsquatting и небезопасные MCP-инструменты.

## Когда читать обязательно

Читать `AGENT_SECURITY.md`, если задача затрагивает:

- `.env`, `.env.example`, ключи, токены, cookies, session strings, credentials;
- Bitrix24, Telegram, MAX, VK, email, webhook, CRM, сделки, лиды;
- Redis/очереди с реальными данными;
- LLM/OpenRouter/Claude/OpenAI или другой внешний AI;
- MCP/API/browser automation, парсинг, scraping, Apify, Playwright;
- публикации, outreach, массовые отправки, рекламу;
- deploy, GitHub, Vercel, production, домены;
- деньги, лимиты, тарифы, подписки.

## Красные линии

Без явного подтверждения нельзя:

- запускать платный API или увеличивать бюджет;
- делать реальные публикации и отправки;
- создавать/изменять реальные CRM-сделки, если это не отдельный разрешённый тест;
- отправлять персональные данные, CRM-данные или секреты во внешние сервисы;
- добавлять скрытые интеграции, telemetry, MCP или фоновые действия;
- делать массовый сбор лидов.

## Зелёный коридор

Можно без отдельного подтверждения:

- читать и править обычные файлы проекта;
- запускать локальные тесты, `compileall`, checker, builder и dry-run;
- создавать локальные отчёты в `data/reports/`;
- работать с синтетическими или обезличенными данными;
- поднимать локальный dev-server, если он ничего не отправляет наружу.

## Применение к агентам

| Агент | Что контролирует security layer |
|---|---|
| Agent 1 Scout | только публичные источники, без серых методов, без закрытых персональных данных |
| Agent 2 Collector | один источник сначала, без массового сбора, с проверкой robots/прав площадки и лимитов |
| Agent 3 Processor | безопасный LLM I/O, защита от prompt injection, structured output, без отправки PII наружу |
| Agent 4 Publisher | нет публикаций без approval, нет неподтверждённых обещаний, нет клиентских данных в контенте |
| Agent 5 CRM | Bitrix24/Telegram только по разрешённому сценарию, секреты не выводить, реальные тесты отдельно подтверждать |
| Agent 6 Outreach | нет спама и массовых отправок, каждое первое касание через approval человека |
| Checker/dashboard | read-only, статусы SET/EMPTY, без Redis/Bitrix24/Telegram/LLM/scheduler/publisher вызовов |

## Чек-лист перед "готово"

Перед финальным ответом по задаче с риском проверить:

- секреты не показаны и не записаны в публичные файлы;
- внешние сервисы не запускались без разрешения;
- платные API не запускались без разрешения;
- реальные публикации, outreach и массовый сбор не запускались;
- данные клиентов обезличены или синтетические;
- результат проверен локальным тестом или dry-run;
- `REPORT.md` обновлён.
