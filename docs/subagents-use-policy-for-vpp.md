# Subagents Use Policy For VPP

Дата: 2026-05-16

Статус: правило работы.

## Коротко

В проекте ВПП постоянных агентов уже 6. Седьмого агента не создаём.

Codex subagents можно использовать как временных помощников для параллельной работы, но они не становятся частью архитектуры проекта.

## Когда subagents нужны

Использовать можно, если задача реально делится на независимые части:

- один subagent анализирует конкурентов;
- второй проверяет лендинг;
- третий готовит варианты контента;
- основной Codex собирает итог и вносит изменения.

## Когда subagents не нужны

Не использовать, если:

- задача маленькая;
- нужен один файл;
- надо сохранить строгий контроль;
- есть риск, что параллельные правки запутают систему;
- работа касается секретов, Bitrix24, публикаций, отправок или платежей.

## Для текущего спринта

Сейчас subagents не нужны.

Причина: мы делаем последовательный controlled sprint на первые 3 заявки. Важнее не скорость, а понятность:

```text
один маршрут
-> один пост
-> один approval
-> одна ручная публикация
-> одна проверка результата
```

## Как они могут пригодиться позже

После approval на расширение можно подключить временных subagents:

| Subagent-задача | Что делает | Что не делает |
|---|---|---|
| Source Explorer | ищет 20 площадок по B2B | не пишет людям |
| Content Critic | улучшает 10 постов | не публикует |
| Landing Reviewer | проверяет посадочные страницы | не деплоит |
| CRM Analyst | анализирует выгрузку Bitrix24 | не меняет CRM |

Все результаты идут через основной оркестратор и `REPORT.md`.
