# Claude Project Rules

Короткие правила проекта, которые можно подключать из `CLAUDE.md` через `@`.

## Секреты и локальные ключи

Применять общий протокол `../PROJECT_SECRET_RULES.md`.

Если Яника передает токен/API-ключ для локальной настройки:

- не повторять его в ответе;
- не писать в `REPORT.md`, `README.md`, `AGENTS.md`, `CLAUDE.md`, `tasks/session-notes.md`, логи или публичные файлы;
- хранить только локально в `.env`, `.env.local`, gitignored `secrets/` или системном хранилище.

Для проверок `.env` показывать только `SET` / `EMPTY`.

Перед GitHub, деплоем, архивом или передачей другому агенту проверить:

- `.env`;
- Redis/Telegram session-файлы;
- Bitrix24 webhooks;
- credentials;
- `logs/`;
- `*.jsonl`;
- `data/*.session`;
- `output/`;
- `.playwright-cli/`.

## Отчетность

Перед началом любой задачи сначала читать `REPORT.md`, чтобы понимать текущее состояние проекта.

После рабочих изменений обновлять `REPORT.md`: что сделано, что проверено, что осталось и следующий маленький шаг.

## Иерархия оркестрации

`CLAUDE.md` — главный оркестратор проекта. Он задаёт правила, роли, архитектуру, порядок исправлений и то, как человеку или AI-ассистенту работать с системой.

`REPORT.md` — память проекта и журнал состояния. Перед задачей его читаем, после изменений фиксируем результат.

`orchestrator/scheduler.py` — технический исполнитель. Он не главнее `CLAUDE.md`; он запускает агентов по расписанию, смотрит очереди и пишет ошибки в лог, когда система реально работает.

Если агент работает неправильно, человек не идёт напрямую “к агенту”. Задача ставится через главный оркестратор проекта: сначала читаем `REPORT.md`, потом по правилам `CLAUDE.md` находим нужный агент/модуль, проверяем логи, исправляем точечно, тестируем и снова фиксируем результат в `REPORT.md`.

## Security Agent Control Layer

`agents/Агент безопастности/AGENT_SECURITY.md` — главный файл управления безопасностью AI-агентов. Он не является седьмым продуктовым агентом и не запускает бизнес-процесс. Это контрольный слой, который применяется ко всем 6 агентам, checker/dashboard, внешним сервисам, LLM и MCP/API.

Когда читать обязательно:

- секреты, `.env`, токены, cookies, session-файлы, webhook URL;
- Bitrix24, Telegram, MAX, VK, email, публикации, outreach;
- платные API, лимиты, подписки, деплой, production;
- реальные клиентские данные, CRM-данные, персональные данные;
- новые MCP/API, браузерные действия, парсинг, внешние LLM-вызовы.

Что он даёт системе:

- красные линии: деньги, продакшен, внешняя коммуникация, секреты и скрытые интеграции только после явного подтверждения;
- зелёный коридор: локальные dry-run, чтение/правка обычных файлов, локальные тесты и синтетические данные;
- AI-защиту: prompt injection, lethal trifecta, SSRF, slopsquatting, markdown/image exfiltration, структурированный LLM I/O;
- чек-лист перед статусом `готово`.

## Правило ремонта агента

1. Перед тем как чинить, прочитать `REPORT.md` и `CLAUDE.md`.
2. Определить, какой агент сломался.
3. Не переписывать всю систему.
4. Проверить логи, `.env`, Redis и очередь этого агента.
5. Починить только минимальную сломанную часть.
6. После проверки обновить `REPORT.md`.

Если визуал ещё нельзя публиковать, использовать нейтральную сгенерированную или схематичную визуализацию без клиентских данных.

## Стек

Python 3.11+ · APScheduler · Redis (AOF включён) · PostgreSQL · dry-run LLM router для локальной разработки · OpenRouter local free test mode · OpenRouter demo-free server mode · OpenRouter production-router server mode · Bitrix24 REST API · Telethon (user account) · python-telegram-bot · vk_api · MAX Bot API.

Сейчас разработка идёт через Codex/подписку, поэтому продуктовый backend держим в `LLM_RUNTIME_MODE=local_codex`, `LLM_PROVIDER=dry_run` и `AGENT_CONTROL_LIVE_LLM=0`.

Для локальной проверки реального ответа модели до сервера есть режим `LLM_RUNTIME_MODE=openrouter_free_test`: все текстовые LLM-задачи идут через одну бесплатную OpenRouter-модель `LLM_MODEL_DEMO_FREE`; Redis, Bitrix24, Telegram, scheduler и publisher от этого не включаются.

Для показа мастеру на сервере есть временный режим `LLM_RUNTIME_MODE=demo_server_free`: все текстовые LLM-задачи идут через одну бесплатную OpenRouter-модель `LLM_MODEL_DEMO_FREE`.

Для финального продукта есть режим `LLM_RUNTIME_MODE=production_server_router`: обычные задачи — DeepSeek Flash/free, управление агентами — MiniMax 2.7, разработка — Opus 4.7, маркетинг — Google Flash. Реальные вызовы включать только после server deploy, проверки model IDs и стоимости.

## Два потока лидов

- Поток А — перепланировки под ключ, вся Россия: B2C, физлица, покупатели, продавцы.
- Поток Б — инженерное проектирование КР/КЖ/КМ/АР/АС/ПОС, стадии П и Р, вся Россия: B2B, застройщики, архитекторы, ГИПы.

Не берём:

- технологические разделы;
- линейные объекты;
- одиночный раздел без комплекса.

## Критические правила

- Агент 6: `approver` должен быть запущен всегда. Нет ответа 30 мин — лид не отправляется + алерт в TG.
- Claude API: hard cap токенов в день в оркестраторе. 80% — алерт. 100% — стоп всех Claude-вызовов.
- Redis: AOF persistence включён. Без него потеря лидов при перезапуске.
- Avito: ротация User-Agent + задержки. 403/captcha — алерт + субагент стоп.
- Blocklist: конкуренты/боты/спамеры — `shared/redis_client.py` blocklist. Фильтрация на входе в Агент 2.
- Контент: разные каналы = разный угол подачи. Не копировать пост из MAX в TG без адаптации.
- Profi.ru: подключение через Playwright с сохранённой сессией от рабочего аккаунта ВПП.
