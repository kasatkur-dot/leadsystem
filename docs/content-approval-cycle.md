# Цикл согласования контента

Дата: 2026-05-06

Статус: управленческий документ. Новые агенты не создавались. PostMyPost, автопубликация, платные API и реальные публикации не запускались.

## 1. Зачем нужен этот цикл

Последняя фото-схема урока 5 показывает, что контентная система должна быть не просто генератором постов.

Нам нужен полный месячный цикл:

```text
foresight -> исследование -> контент-план -> продакшен -> согласование -> перегенерация -> публикация -> аналитика -> следующий месяц
```

Простыми словами:

1. Сначала решаем, какую гипотезу месяца проверяем.
2. Потом собираем факты, боли, темы и источники.
3. Потом делаем план публикаций.
4. Потом Agent 4 готовит черновики.
5. Потом человек согласует.
6. Если плохо, Agent 4 переделывает.
7. Только после согласования можно публиковать.
8. Agent 5 собирает цифры.
9. На следующий месяц берём не случайные идеи, а выводы из аналитики.

## 3. Чего не хватает Agent 4

Agent 4 уже умеет делать контент, но ему не хватает управленческих входов и статусов.

Нужно добавить не нового агента, а правила и шаблоны:

| Этап | Чего не хватает | Как закрываем сейчас |
|---|---|---|
| `foresight` | Нет короткой гипотезы месяца | Поле в месячном плане |
| `исследование` | Agent 1 пока не передаёт Agent 4 готовые темы | Поле `research_source` и `audience_pain` в плане |
| `контент-план` | Нет единой таблицы на месяц | `content/pipeline/monthly-content-plan-template.md` |
| `продакшен` | Есть генерация, но не привязана к месячной цели | Поле `goal` для каждой единицы контента |
| `согласование` | Нет статуса review/approved | Добавляем статусный путь |
| `перегенерация` | Нет явной причины правки | Добавляем поля `review_notes` и `regeneration_notes` |
| `публикация` | Реальные публикации технически возможны, но рано | Оставляем manual/dry-run до подтверждения |
| `следующий месяц` | Нет обратной связи из аналитики | Agent 5 заполняет ретроспективу |

### Роли Agent 4 как этапы, а не новые агенты

По проверке ДЗ 4 контентный продакшен нельзя делать одним хаотичным промптом. Но для нашего MVP это не повод создавать новых агентов.

Внутри Agent 4 держим этапы:

```text
idea-research -> content-plan -> text/script -> visual-brief -> storyboard -> generation -> QA/approval
```

Что это значит простыми словами:

- сначала понять идею и источник;
- потом поставить её в месячный план;
- потом написать текст или сценарий;
- потом отдельно описать визуал;
- для будущего видео отдельно сделать storyboard;
- потом только генерировать;
- перед публикацией обязательно согласовать.

Статус шага: закрыт документально. Новые агенты не создавались.

## 5. Статусы контента

Используем один простой путь:

```text
idea -> planned -> draft -> review -> needs_changes -> approved -> scheduled -> published -> analyzed
```

Дополнение от 2026-05-16:

Перед статусом `approved` пост должен быть привязан к воронке и иметь поля из `content/pipeline/post-funnel-operating-system.md`:

```text
Funnel arc
Funnel stage
Offer ID
Offer readiness
CTA intensity
Next bridge
Metric to watch
```

Если этих полей нет, пост может оставаться черновиком, но не считается системной публикацией lead-engine.

Проверка контракта:

```bash
.venv/bin/python scripts/check_content_funnel_contract.py
```

Скрипт только читает локальные файлы и пишет `data/reports/content_funnel_contract_check.json`.

Что значит каждый статус:

| Статус | Значение |
|---|---|
| `idea` | Есть идея, но ещё нет решения публиковать |
| `planned` | Идея попала в месячный план |
| `draft` | Agent 4 сделал черновик |
| `review` | Человек проверяет |
| `needs_changes` | Нужно переделать |
| `approved` | Можно готовить к публикации |
| `scheduled` | Запланировано вручную или через будущий сервис |
| `published` | Опубликовано и записано в журнал |
| `analyzed` | Agent 5 или человек разобрал результат |

## 6. Правила согласования

Перед публикацией проверяем:

- Текст подходит аудитории канала.
- Есть польза, а не просто реклама.
- Нет обязательной формы, квиза, email или телефона.
- Нет обещаний результата, которые нельзя гарантировать.
- Нет персональных данных клиентов.
- Есть мягкий CTA: написать вопрос, в личные сообщения или комментарии.
- Каналы не дублируют один и тот же текст.

Если не проходит:

```text
status = needs_changes
review_notes = что исправить простыми словами
regeneration_notes = что Agent 4 должен сделать иначе
regeneration_reason = короткая причина: tone / facts / duplicate / CTA / legal_risk / visual / other
```

## 8. PostMyPost и автопубликация

PostMyPost остаётся отложенным.

Актуальный план полной автоматизации выпуска и публикации:

```text
docs/content-autopublishing-automation-plan.md
```

Первый безопасный технический шаг: `approved-only dry-run` для MAX. Система должна взять только утвержденный пост, проверить контракт, проверить канал и остановиться перед реальной отправкой.

Порядок подключения позже:

1. Сначала 1 месяц ручного цикла.
2. Потом 3-5 dry-run публикаций без внешней отправки.
3. Потом 1 реальная публикация в одном канале после подтверждения.
4. Только потом PostMyPost или другой автопостинг.

Что нельзя сейчас:

- Не запускать автопубликацию.
- Не подключать PostMyPost токены без отдельного решения.
- Не публиковать пачкой во все каналы.
- Не запускать контент-ферму отдельно от lead-engine.

## 9. Где позже нужен Sentry/logging

Sentry/logging нужен не для оценки красоты текста, а чтобы понять, где технически сломалась цепочка.

Позже логирование нужно поставить на такие места:

| Место | Что логировать | Зачем |
|---|---|---|
| Agent 1 -> Agent 4 | research card создана или не создана | Понять, дошла ли разведка до контент-плана |
| Agent 4 generation | provider, model, dry_run, cost, error | Видеть стоимость и ошибки генерации |
| approval step | status, reason, approved_by | Понимать, почему контент застрял |
| publication adapter | dry_run/published/error | Не потерять ошибку публикации |
| Redis `content_published` | событие создано или нет | Проверить связь Agent 4 -> Agent 5 |
| Agent 5 analytics | отчет создан или ошибка | Проверить закрытие цикла аналитикой |

Правило на будущее:

```text
LLM анализирует только нужный фрагмент ошибки, а не все логи проекта целиком.
```

Статус шага: закрыт документально. Sentry не подключался.
