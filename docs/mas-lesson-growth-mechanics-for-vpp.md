### 1.1. Механика роста до подписчиков и заявок

В `МАС` есть playbook:

```text
MVP -> контент-завод -> реклама -> заявки
```

Шаги из проекта:

```text
выбрать продуктовую гипотезу и аудиторию
-> собрать MVP или понятный оффер
-> создать лендинг с регистрацией/заявкой
-> собрать 20-30 коротких единиц контента
-> запустить тестовые рекламные связки
-> передавать заявки в CRM/таблицу/Supabase
-> считать CPL, CR, CAC, оплаты и удержание
-> масштабировать только работающие связки
```

Отдельно в артефактах `МАС` есть задача:

```text
Запусти новый онлайн-продукт по подписке: нужен лендинг, контент-завод,
реклама, 8000 подписчиков, заявки, CRM, аналитика и деплой на сервер.
```

Важно: `8000 подписчиков` зафиксировано как цель задачи/артефакта. В полной транскрибации `Лекция_7.md` найден практический механизм поиска источников подписчиков через TGStat/телега-аналитику и рекламные закупки конкурентов. Поэтому для ВПП берем не обещание цифры, а управляемый процесс: `канал-донор -> UTM -> подписчик -> вопрос -> исходные данные -> Bitrix24 -> КП/договор`.

### 1.2. Как там устроен контент-завод

Из артефакта `Контент-завод`:

```text
хук
-> проблема
-> объяснение
-> пример
-> CTA
```

Ритм:

- 20-30 коротких единиц контента на первую волну;
- 2-4 коротких видео в день на тестовом этапе;
- 1-2 поста в день в основном канале;
- лучшие темы и хуки превращаются в рекламные креативы.

Для ВПП это нельзя копировать буквально каждый день без ресурсов. Но можно взять принцип:

```text
много коротких проверок спроса
-> измерение реакции
-> усиление тем, где есть вопросы и заявки
```

### 1.3. Как там устроена реклама

Из артефакта `Рекламные связки и тесты`:

- запускать только связку `аудитория + оффер + креатив + посадочная`;
- на старте минимум `3 аудитории x 3 креатива`;
- обязательно UTM;
- слабые связки отключать после данных;
- смотреть CTR, CPC, CPL, CR посадочной и качество лида.

Для ВПП это значит:

```text
не запускать рекламу "на всю Россию вообще"
-> сначала 3 сегмента, 3 оффера, 3 креатива
-> каждый лид падает в Bitrix24 с источником и сегментом
```

### 1.4. AI-астропсихолог как продуктовая модель

В `Лекция_5.md` найден пример продукта:

- астрологический / астропсихологический AI-продукт;
- уже протестирован через Telegram;

```text
-> Telegram/чат как быстрый MVP
-> база знаний/исходные данные
-> тест на реальных людях
-> улучшение промпта и модели
-> приложение/лендинг/личный кабинет позже
```

## 2. Как это применить к ВПП

### 2.1. Наш аналог AI-астролога

Нам нужен не `AI-астролог`, а продуктовые AI-помощники под проектную организацию.

| AI-продукт ВПП | Что делает | Куда ведет |
|---|---|---|
| AI-проверка исходных данных | говорит, хватает ли данных для КП за 24 часа | Bitrix24 сделка |
| AI-помощник перепланировки | собирает план, БТИ, задачу, тип помещения, риски | консультация / проект |
| AI-помощник коммерческого объекта | уточняет площадь, стадию, ПД/РД, экспертизу, сроки | B2B-сделка |
| AI-помощник реконструкции | отделяет простую заявку от сложного объекта | инженерная проверка |
| AI-помощник тендера | вытаскивает из ТЗ сроки, состав работ, требования и риски | решение: участвовать или нет |

Это должны быть не отдельные агенты, а сценарии внутри текущих 6 агентов.

### 2.2. Наша growth-цепочка для заказчиков

Правильная цепочка:

```text
сегмент
-> оффер
-> лендинг / пост / карточка
-> короткий полезный разбор
-> запрос исходных данных
-> Bitrix24
-> КП / звонок / инженерная проверка
-> договор
-> ROMI-отчет
```

Для первого теста:

```text
коммерческая перепланировка
склады / ангары
реконструкция
```

Квартиры и дома сохраняем, но B2B идет первым приоритетом.

### 3.3. Рекламные связки ВПП

Первая тестовая структура:

| Сегмент | Аудитория | Оффер | Креатив |
|---|---|---|---|
| склады/ангары | собственники, девелоперы, производство | состав ПД/РД и КП после исходных данных | чеклист исходных данных |
| коммерческая перепланировка | арендаторы, магазины, офисы, ТЦ | проект для арендодателя/УК/согласования | до ремонта проверьте проектный путь |
| реконструкция | собственники коммерческих объектов, УК | первичный разбор состава работ | что нельзя оценить без обследования |

Запускать только после готовности:

- лендинг;
- UTM;
- Bitrix24 поля;
- ответственный менеджер;
- dashboard;
- бюджет и стоп-правила.

### 3.4. Метрики

Считать не просто подписчиков.

Главные метрики ВПП:

| Уровень | Метрики |
|---|---|
| Контент | охват, сохранения, вопросы, переходы |
| Подписчики | прирост, источник, качество, вопросы от новых |
| Лендинг | визиты, клики CTA, обращения |
| AI intake | сколько людей дали исходные данные |
| Bitrix24 | сделки, сегмент, источник, стадия, ответственный |
| Продажи | лид -> звонок, звонок -> КП, КП -> договор |
| Деньги | выручка, ROMI, CAC, payback |

## 4. Что важно из мини-лекции про кейсы

В расшифровке `20260323_124626.tsv` найден важный разбор:

- кейсы могут быть хорошими, но без упоминания клиента/типа клиента вызывают недоверие;
- если нет понятной связи с выгодой бизнеса, кейс слабее продает;
- нужно показывать не только работу, но и результат: лиды, выручка, бизнес-эффект, если это можно раскрывать.

Для ВПП это значит:

```text
кейс должен отвечать:
кто был клиент без раскрытия персональных данных
какой был объект
какая задача
что сделали
какой измеримый результат можно показать
что было бы риском без проекта
```

Если нельзя раскрыть клиента, пишем честно:

```text
коммерческий объект, город/регион, площадь, тип задачи, без адреса и реквизитов.
```

## 6. Стоп-правила

Нельзя:

- обещать 8000 подписчиков как гарантированный результат;
- обещать большие деньги по аналогии с AI-астрологом;
- запускать рекламу без лендинга, UTM и Bitrix24;
- собирать персональные данные без понятной цели;
- раскрывать адреса, ФИО, телефоны, суммы, штампы, кадастровые номера и служебные отметки;
- делать AI-продукт вместо основной услуги проектирования.

Можно:

- использовать механику быстрого MVP;
- проверять спрос через контент и микро-разборы;
- делать AI-помощника как вход в услугу;
- считать путь от подписчика до сделки;
- масштабировать только то, что дает качественные обращения.
