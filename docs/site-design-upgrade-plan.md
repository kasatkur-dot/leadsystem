# План дизайн-апгрейда сайта Вектор Плюс-Про

Дата: 2026-05-10.

Статус: внутренний план. Сайт не менять в рамках этого документа. Публикацию, Vercel, GitHub deploy, CRM, Bitrix24, внешние формы и платные генерации не запускать без отдельного подтверждения.

## 1. Зачем нужен этот план

Текущий сайт уже собрал базовую структуру: услуги, доверие, кейсы, процесс, FAQ, медиа, о компании и контакты. Но по визуальному уровню он всё ещё дорабатывается кусками. Чтобы не чинить бесконечно один блок за другим, нужен единый дизайн-план.

Цель апгрейда:

- сделать сайт красивым, современным и уверенным;
- убрать ощущение черновика и внутренней памятки;
- усилить доверие через факты, документы, кейсы и понятный процесс;
- сохранить SEO/GEO/AI-видимость;
- не копировать чужие сайты, изображения и тексты;
- не раскрывать клиентские данные.

## 2. Почему Remplanner учитывать, но не копировать

Remplanner полезен как референс не потому, что его нужно повторять визуально, а потому что он хорошо показывает логику сервиса:

- быстро объясняет, что получает клиент;
- показывает путь от запроса к результату;
- использует визуальные примеры;
- делает сложную услугу понятной;
- ведёт к действию без лишней технической нагрузки.

Для Вектор Плюс-Про нужно взять принцип:

```text
услуга -> понятный результат -> примеры -> доверие -> быстрый контакт
```

Нельзя брать:

- чужую структуру один в один;
- чужие тексты;
- чужие изображения;
- визуальные решения, которые делают сайт похожим на сервис перепланировок, если нам нужен шире профиль проектной организации.

## 3. Три подходящих дизайн-направления

### Направление 1. Премиальное инженерное бюро

Смысл: сайт выглядит как сильная проектная организация, которой можно доверить сложный объект.

Визуал:

- спокойный светлый фон;
- тёмный инженерный синий;
- крупные, но не чрезмерные заголовки;
- реалистичные фото зданий, конструкций, чертежей;
- аккуратные карточки документов;
- минимум декоративных плашек.

Подходит для:

- проектной документации;
- рабочей документации;
- реконструкций;
- коммерческих и общественных объектов;
- клиентов, которым важны СРО, ответственность и документы.

Риск: если сделать слишком строго, сайт станет сухим и похожим на корпоративный PDF.

### Направление 2. Понятный сервис проектных решений

Смысл: сайт объясняет сложные проектные услуги простым языком и ведёт клиента по шагам.

Визуал:

- сильный первый экран с понятной пользой;
- блоки “что нужно подготовить”, “что получите”, “как проходит работа”;
- карточки услуг с человеческими сценариями;
- FAQ по болям клиента;
- мягкая анимация пути клиента.

Подходит для:

- перепланировок;
- частных клиентов;
- предпринимателей;
- людей, которые не знают, какие документы нужны.

Риск: если уйти только в простоту, можно потерять образ серьёзной проектной организации.

### Направление 3. Цифровая проектная система

Смысл: сайт показывает, что Вектор Плюс-Про работает не хаотично, а как система: исходные данные, состав работ, документы, сопровождение, аналитика заявок.

Визуал:

- интерфейсные блоки;
- схемы маршрутов;
- аккуратные dashboard-элементы;
- связка “заявка -> проверка -> проектное решение -> документы”;
- больше структуры, меньше декоративности.

Подходит для:

- B2B-заказчиков;
- генподрядчиков;
- бюджетных и коммерческих объектов;
- будущей связи сайта с CRM/Bitrix24 и аналитикой.

Риск: если переборщить с интерфейсностью, сайт будет выглядеть как внутренняя админка, а не продающий лендинг.

## 4. Рекомендуемый выбор

Для первой сильной версии лучше объединить:

```text
70% Премиальное инженерное бюро
20% Понятный сервис проектных решений
10% Цифровая проектная система
```

Простыми словами: внешний вид должен быть как у сильной проектной компании, а объяснение услуг — как у понятного сервиса.

## 5. Инструменты

### Refero / getdesign

Задача: получить дизайн-токены и визуальные правила по референсам.

Использовать для:

- цветовой системы;
- типографики;
- сетки;
- карточек;
- кнопок;
- отступов;
- DESIGN.md-спецификации.

Ограничение: не копировать чужой дизайн. Использовать только как источник правил и визуального направления.

### Relume

Задача: проверить структуру сайта и сценарии страниц.

Использовать для:

- sitemap;
- wireframe;
- порядка блоков;
- логики CTA;
- проверки, не забыли ли мы важные секции.

Ограничение: не переносить шаблон вслепую. Для проектной организации нужны свои тексты и доказательства.

### shadcn / Aceternity

Задача: взять качественные паттерны компонентов.

Можно брать идеи для:

- hero;
- bento-блоков;
- FAQ;
- карточек услуг;
- карточек кейсов;
- tabs/accordion;
- trust-панели;
- CTA-блоков.

Ограничение: текущий сайт статический HTML/CSS/JS. Полный переход на React/Next.js делать только отдельным решением. Сейчас можно переносить паттерн вручную, без большой миграции.

### imagegen

Задача: создать безопасные визуалы без клиентских данных.

Использовать для:

- реалистичных зданий;
- инженерных/конструктивных сцен;
- нейтральных чертежей;
- обложек кейсов;
- промо-изображений для MAX/Telegram/VK/Дзен;
- social preview.

Ограничение: не выдавать сгенерированные изображения за реальные объекты компании. Если визуал сгенерирован, внутри проекта фиксировать это как “безопасная иллюстрация”.

### Browser / Playwright

Задача: проверять сайт как посетитель.

Использовать для:

- desktop-проверки;
- mobile-проверки;
- кликабельности CTA;
- hash-ссылок;
- скриншотов;
- переполнений текста;
- визуального QA после каждой маленькой партии правок.

Ограничение: Browser Use в Codex может быть ограничен политикой in-app browser. Если он не видит локальный сайт, использовать Playwright/скриншоты или проверку глазами пользователя.

### Готовые репозитории и шаблоны

Можно рассматривать только как источник паттернов, а не как готовую замену сайта.

Кандидаты:

- shadcn/ui templates;
- Aceternity UI blocks;
- Magic UI blocks;
- Tailwind landing templates;
- open-source Vite/React landing templates.

Когда использовать:

- если решим перейти с текущего статического сайта на React/Vite или Next.js;
- если нужен современный компонентный слой;
- если текущий HTML станет тяжело поддерживать.

Когда не использовать:

- прямо сейчас без решения о миграции;
- если шаблон ломает SEO/GEO-структуру;
- если шаблон требует внешних сервисов, платных компонентов или сложной сборки.

## 6. Многоагентный процесс

```text
Scout -> Creator -> Codex -> QA -> Analytics
```

### Agent 1 Scout

Что делает:

- собирает референсы сайтов проектных организаций;
- анализирует Remplanner как референс наполнения;
- собирает конкурентов;
- отмечает сильные UX-паттерны;
- фиксирует, что можно взять как принцип, а что копировать нельзя.

Результат:

- список референсов;
- выводы по структуре;
- список удачных блоков;
- риски копирования.

### Agent 4 Publisher / Creator

Что делает:

- переписывает тексты сайта простым языком;
- усиливает экспертность;
- готовит FAQ по болям клиентов;
- делает карточки услуг;
- готовит безопасные кейсы;
- готовит медиа-упаковку.

Результат:

- тексты для hero;
- услуги;
- кейсы;
- FAQ;
- медиа-блоки;
- CTA.

### Codex

Что делает:

- внедряет дизайн и тексты в локальный сайт;
- правит HTML/CSS/JS;
- не подключает формы и внешние сервисы без подтверждения;
- сохраняет SEO/GEO-структуру;
- проверяет отсутствие секретов.

Результат:

- локальная рабочая версия сайта;
- обновлённые файлы;
- запись в REPORT.md.

### QA

Что делает:

- проверяет desktop;
- проверяет mobile;
- проверяет клики;
- проверяет CTA;
- проверяет readable text;
- проверяет alt, focus-visible, hash-ссылки, отсутствие переполнений.

Результат:

- список критичных проблем;
- только после этого разрешение на следующую волну.

### Agent 5 CRM / Analytics

Что делает позже:

- смотрит, какие CTA ведут к заявке;
- связывает источник трафика, страницу, заявку, сделку и ROMI;
- помогает понять, какие блоки сайта реально дают лиды.

В первой версии не подключать формы, Bitrix24 и сквозную аналитику без отдельного подтверждения.

## 7. Что менять первой волной

Первая волна — без полной переделки сайта.

Сделать:

- выбрать одно дизайн-направление;
- обновить `site/DESIGN.md` под выбранный стиль;
- привести первый экран к одной сильной композиции;
- убрать лишние плашки и повторяющиеся блоки;
- сделать услуги понятнее: когда нужна услуга, что получает клиент, какие исходные данные нужны;
- усилить доверие: СРО, ИНН/ОГРН, Яндекс Карты, 2GIS, MAX-канал, безопасные кейсы;
- сделать FAQ по реальным страхам клиента;
- проверить mobile.

Не делать:

- деплой;
- формы;
- CRM;
- платные генерации;
- полную миграцию на React/Next.js.

## 8. Что менять второй волной

Вторая волна — усиление наполнения и визуалов.

Сделать:

- подготовить 3-5 безопасных кейсов;
- заменить слабые визуалы на разные по смыслу изображения;
- сделать блок “примеры результата”: ПД, РД, КР, КЖ, КМ без раскрытия клиентских данных;
- подготовить медиа-блок: короткое видео, схема пути клиента, обложки для каналов;
- добавить страницы услуг первой волны;
- проверить SEO/GEO-тексты по каждой услуге.

## 9. Что оставить на потом

Позже:

- React/Next.js или Vite-миграция, если статический сайт станет тесным;
- shadcn/Aceternity-компоненты как полноценная система;
- HyperFrames/Remotion/HeyGen для промо-видео;
- Vercel preview;
- подключение домена;
- формы;
- Bitrix24;
- сквозная аналитика;
- A/B-тесты CTA;
- ROMI по источникам.

## 10. Почему сайт не получился идеальным сразу

Причины:

- старт был не с финального дизайн-макета, а с кода, выгруженного из Claude Design;
- требования уточнялись по ходу: сначала структура, потом визуалы, потом Remplanner, потом SEO/GEO, потом контакты, потом карточки, потом медиа;
- часть реальных визуалов нельзя было безопасно вставить без обезличивания;
- нельзя было копировать чужой сайт и чужие изображения;
- мы сознательно не запускали платные сервисы, деплой, формы и CRM без подтверждения;
- я делал много точечных правок по скринам, а надо было раньше остановиться и собрать единый дизайн-спринт.

Честный вывод:

```text
Сайт не плохой как черновая рабочая основа, но хороший продающий сайт требует не только кода, а отдельного дизайн-процесса: референсы -> дизайн-система -> тексты -> визуалы -> QA -> только потом публикация.
```

## 11. Один ближайший маленький шаг

Следующий шаг:

```text
Выбрать одно из трёх дизайн-направлений и обновить site/DESIGN.md под него.
```

Рекомендуемый выбор:

```text
Премиальное инженерное бюро + понятный сервис проектных решений.
```

После выбора можно делать первую безопасную партию правок сайта: только hero, услуги и доверие, без деплоя и без подключения внешних сервисов.

## 12. Актуальное рабочее ТЗ

После уточнения логики первого экрана и Lazyweb-аудита актуальное ТЗ вынесено в отдельный файл:

```text
docs/current-site-improvement-tz.md
```

Этот файл считать главным рабочим источником для следующей партии доработок сайта.

Ключевое дополнение:

- посетитель не читает сайт как книгу, а сканирует первый экран;
- первый экран должен сразу ответить, что делает компания, кому помогает, какой результат даст, почему ей можно доверять и куда нажать;
- услуги, кейсы, процесс, FAQ и CTA нужно дописывать не как внутреннюю памятку, а как понятный путь клиента от проблемы к первому безопасному шагу.
