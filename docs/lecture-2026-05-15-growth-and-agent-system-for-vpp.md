## 1. Что лектор показал

### 1.1. Многоагентная система как операционный центр

Таймкод: `00:33:22-00:35:42`.

Лектор показывает, что через Telegram/оркестратор можно дать команду создать многоагентную систему. В примере система быстро создает:

- внутри департаментов роли, правила, инструменты, доступы, KPI.

Главная мысль для ВПП: нам не надо плодить 27-30 агентов. Нужно оставить текущие 6 агентов, но добавить поверх них понятные сценарии и правила: кто ищет источник, кто делает контент, кто квалифицирует, кто пишет в CRM, кто считает результат.

### 1.2. Пример AI-астрологии

Таймкод: `00:39:24-00:43:32`.

По словам лектора:

- команда выпускает до 2 продуктов в неделю, когда процесс уже автоматизирован;
- приложение по астрологии заработало `9,5 млн рублей за 3 месяца`;
- чистое время запуска было около `1,5 недели`;

- была связка `продукт -> контент-завод -> реклама -> заявки`;

- путь клиента: короткий контент -> профиль -> сайт -> короткая регистрация -> приложение.

Что берем для ВПП: не астрологию и не обещание денег, а саму механику. Экспертность ВПП уже есть: СРО, 10+ лет, проектирование, ПД/РД, перепланировки, реконструкции. Нам нужен такой же продуктовый вход:

```text
полезный короткий контент
-> доверие к экспертности
-> страница/пост по сегменту
-> AI-проверка исходных данных
-> Bitrix24
-> КП за 24 часа только если данных достаточно
-> договор
```

### 1.4. Фронтенд и бекенд должны быть связаны

Таймкод: `01:16:24-01:20:00`.

Лектор объясняет, что визуальный интерфейс сам по себе бесполезен, если он не связан с backend-логикой агентов. Нужно соединить:

```text
backend агента
-> сценарии
-> данные
-> frontend
-> управление через интерфейс
```

Для ВПП это значит: dashboard должен показывать не красивые блоки, а живые рабочие состояния:

- какие источники найдены;
- какие посты готовы;
- какие лиды квалифицированы;
- какие заявки требуют данных;
- какие сделки ушли в Bitrix24;
- где есть дубли;
- где ROMI положительный/отрицательный.

### 1.5. Самостоятельный продукт и OpenRouter

Таймкод: `01:19:36-01:20:00`.

Лектор говорит, что если продукт должен работать сам, не через подписку Cloud/Codex, ему нужны собственные модели через OpenRouter или аналогичный слой. Для ВПП это этап позже. Сейчас система локальная и безопасная; платные модели и deploy не включаем без отдельного решения.

### 1.6. Экономия на моделях и embeddings

Таймкод: `01:34:29-01:40:12`.

Главная мысль: не надо дорогой сильной моделью рыться во всей базе знаний. Для больших документов и архивов нужны embeddings/поисковый слой, а сильная модель должна получать уже найденные фрагменты.

Для ВПП это важно из-за:

- проектных документов;
- старых заявок;
- Bitrix24;
- кейсов;
- СРО/регламентов;
- будущих лендингов и материалов.

Правильная схема:

```text
документы и база знаний
-> дешевый поиск / embeddings
-> найденные фрагменты
-> сильная модель только для ответа, КП, анализа риска
```

### 1.7. Сервер, безопасность, Supabase

Таймкоды: `02:05:53-02:10:01`, `02:12:35-02:13:39`.

Лектор объясняет:

- серверный продукт изолирован от личного компьютера;
- после deploy нужен свой orchestrator/admin-файл с правилами системы;
- можно добавлять инструменты: лендинги, презентации, интернет-поиск, парсинг, посты, соцсети, content factory;
- Supabase не сервер, а база/хранилище;
- Supabase нужен, когда есть пользователи, регистрации, документы, переписки, большой контекст.

Для ВПП: deploy и Supabase нужны позже, когда появятся реальные пользователи личного кабинета или публичный AI-помощник. Пока достаточно локальных файлов, dashboard, Bitrix24 и read-only аналитики.

## 2. Методика лектора в виде простой схемы

```text
1. Создать жесткие правила безопасности.
2. Описать многоагентную оргструктуру.
3. Сделать продукт или понятный вход в услугу.
4. Создать контент-завод.
5. Найти много источников аудитории.
6. Запустить трафик или outreach.
7. Принять подписчиков/лиды.
8. Включить прогрев 3-4 касания в день.
9. Вести человека в продукт/заявку.
10. Считать деньги, CPL, конверсию, ROMI.
11. Оптимизировать источники и стоимость.
```

Для ВПП этот же принцип должен выглядеть так:

```text
1. Безопасность и ручное подтверждение опасных действий.
2. 6 текущих агентов, без создания Agent 7.
3. Сегментный продуктовый вход: склад, коммерческая перепланировка, реконструкция, ТЦ/магазин, квартира, дом.
4. Контент-завод на экспертных разборах.
5. Публичные источники аудитории и B2B-площадок.
6. Только approved outreach/реклама.
7. Входящий вопрос или заявка.
8. AI-менеджер собирает исходные данные.
9. Bitrix24 фиксирует источник, сегмент, готовность к КП.
10. Dashboard считает заявки, КП, договоры, деньги.
11. Система усиливает только работающие источники.
```

## 3. Как применить к нашему проекту

### 3.1. Что делать первым

Первым нужно строить не рекламу, а `source intelligence`:

```text
публичный источник
-> тип аудитории
-> сегмент ВПП
-> качество
-> безопасный способ касания
-> возможный контент
-> UTM/метка
-> результат в Bitrix24
```

Начальные источники:

- MAX-канал `СИЛА Проекта`;
- Telegram/VK-каналы про коммерческую недвижимость, ремонт бизнеса, склады, ТЦ, аренду;
- форумы и сообщества предпринимателей;
- Яндекс Карты и 2GIS для карточек и отзывов;
- Avito/Profi/Яндекс Услуги как вторичный поток;
- тендерные и коммерческие площадки;
- сайты девелоперов, УК, ТЦ, складских операторов;
- старые Bitrix24-контакты и сделки.

### 3.2. Наш аналог “AI-астролога”

Для ВПП самый сильный аналог не развлекательный, а прикладной:

```text
AI-помощник проверки готовности к КП
```

Он должен отвечать:

- какой объект;
- кто клиент: юрлицо, ИП, физлицо;
- город;
- площадь;
- назначение помещения;
- есть ли планы БТИ/обмеры/ТЗ;
- нужна ли ПД, РД, перепланировка, реконструкция, экспертиза;
- хватает ли данных для КП за 24 часа;
- чего не хватает;
- какой следующий шаг.

Это уже частично сделано в dry-run AI-менеджера. Следующий шаг: связать его с growth-источниками.

### 3.3. Контент-завод ВПП

Формула поста:

```text
крючок
-> ошибка или риск
-> простой пример
-> какие исходные данные нужны
-> мягкий переход к разбору
```

Темы первой волны:

- почему КП за 24 часа возможно только при полном комплекте данных;
- что юрлицу нужно для проекта перепланировки;
- какие документы нужны для склада/ангара;
- чем ПД отличается от РД;
- когда нужна экспертиза;
- почему нельзя считать проект только по площади;
- что проверять до аренды помещения под магазин/офис;
- что скрывается в ТЗ от ТЦ/арендодателя;
- как понять, что реконструкция сложнее обычной перепланировки.

### 3.4. Метрики

Лектор смотрит подписчиков, просмотры, заявки и деньги. Для ВПП нужно считать не только подписчиков:

| Метрика | Зачем |
|---|---|
| source_id | понимать, откуда пришел человек |
| segment | склад, коммерция, квартира, дом, реконструкция |
| useful_subscriber | подписчик из нужной аудитории |
| question_count | сколько людей задают вопросы |
| source_data_received | сколько прислали исходные данные |
| kp_ready | хватает ли данных для КП |
| kp_sent | сколько КП отправлено |
| deal_created | сколько сделок в Bitrix24 |
| contract_won | сколько договоров |
| revenue | выручка |
| ROMI | что масштабировать |

Главная метрика для ВПП:

```text
стоимость одного лида с исходными данными, достаточными для КП
```

А не просто подписчик.

## 5. Решение

Внедрять нужно так:

1. Зафиксировать расшифровку и методику.
2. Создать локальную карту источников без внешних запусков.
3. Связать карту источников с AI-manager dry-run.
4. Добавить dashboard-блок по growth-методике.
5. Подготовить 10-20 безопасных источников-кандидатов.
6. Подготовить 5 постов под B2B-сегменты.
7. После ручного approval Яники сделать один маленький тест: один пост, один источник, один UTM, без массовой рассылки.

Это переносит механику лектора в ВПП, но без опасных действий и без обещания невозможных цифр.
