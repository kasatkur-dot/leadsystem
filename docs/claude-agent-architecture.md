# Claude Agent Architecture

Архитектурная карта агентов проекта `design-studio-lead-engine`.

Этот файл вынесен из `CLAUDE.md`, чтобы главный файл оставался коротким.

## Правило обновления

Если агент улучшается или получает новый функционал, обновить его файл правил в `agents/`.

Если изменение влияет на карту ролей, цепочку агентов или порядок запуска, обновить этот файл и кратко зафиксировать изменение в `REPORT.md`.

## Главная логика

- `CLAUDE.md` — главный оркестратор проекта по правилам и смыслу.
- `orchestrator/` — техническая папка с Python-запускателем, не уровень выше `CLAUDE.md`.
- `agents/` — шесть агентских отделов проекта.
- Не все агенты обязаны быть уже подключены к runtime-расписанию. Часть папок — архитектурная заготовка под следующие волны.

## Управленческая иерархия

```text
CLAUDE.md — главный оркестратор проекта
│
├── REPORT.md — память и журнал состояния
│
├── orchestrator/scheduler.py — технический запускатель MVP-цепочки
│
└── agents/ — 6 агентских отделов проекта
```

## Что реально запускает `orchestrator/scheduler.py`

```text
orchestrator/scheduler.py
│
├── Агент 2 / tender_collector       каждые 15 минут
├── Агент 3 / processor              каждые 5 минут
├── Агент 5 / crm_router             каждые 5 минут
├── Агент 6 / relevance_pipeline     каждые 2 минуты
├── Агент 6 / tg_monitor             фоновый поток
├── Агент 6 / approver               фоновый поток
└── Агент 6 / sender                 фоновый поток
```

Агент 1 есть в полной архитектуре и файловой структуре, но сейчас не подключён к MVP-расписанию.

Агент 4 перенесён внутрь lead-engine как контент-движок, но его реальные публикации запускаются вручную через CLI до заполнения publisher-токенов и проверки каналов.

## Shared-модули

- `shared/models.py` — RawLead, QualifiedLead, OutreachLead.
- `shared/redis_client.py` — очереди raw/qualified/outreach/content_events + `blocklist`.
- `shared/db.py` — PostgreSQL: LeadRecord, DuplicateFingerprint, AgentLog.
- `shared/logger.py` — единый логгер.
