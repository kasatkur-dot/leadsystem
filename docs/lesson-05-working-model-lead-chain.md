# Применение рабочей модели урока 05.05 к lead-engine

Дата: 2026-05-07.

Статус: внутренний документ. Новые агенты не создавались, код не запускался, платные сервисы и реальные публикации не использовались.

## 1. Модель из лекции

Рабочая модель урока:

```text
Главный оркестратор
-> департаменты
-> агенты внутри департаментов
-> скиллы/API/MCP у конкретных агентов
-> маленький тест
-> отчет
-> масштабирование только после результата
```

Важная продажная связка:

```text
новый лид
-> квалификация
-> персональное КП или лендинг
-> CRM / карточка клиента
-> следующий шаг
-> аналитика
```

## 2. Текущая система как департаменты

| Департамент | Что делает | Где в проекте |
|---|---|---|
| Оркестрация | задаёт правила, порядок и запуск | `CLAUDE.md`, `REPORT.md`, `orchestrator/scheduler.py` |
| Разведка | ищет источники, конкурентов, спрос и рыночные сигналы | `agents/agent1_scout/` |
| Сбор лидов | собирает сырые лиды и сигналы спроса | `agents/agent2_collector/`, часть `agents/agent6_outreach/` |
| Квалификация | чистит, обогащает, скорит лид и предлагает следующий шаг | `agents/agent3_processor/` |
| Упаковка / КП / контент | готовит офферы, контент, лендинги и точки доверия | `agents/agent4_publisher/`, `site/`, частично `agents/agent3_processor/offer_gen/` |
| CRM и контроль | создаёт карточку, уведомляет менеджера, держит следующий шаг | `agents/agent5_crm/` |
| Аутрич / первое касание | отвечает в чатах после одобрения и передаёт заинтересованных в CRM | `agents/agent6_outreach/` |
| Аналитика | считает каналы, расходы, лиды, сделки, ROMI | `agents/agent5_crm/analytics_reporter/` |

## 3. Связка `новый лид -> квалификация -> КП/лендинг -> CRM -> следующий шаг`

| Шаг | Учтено сейчас | Где именно | Статус |
|---|---|---|---|
| Новый лид | Да | `RawLead`, `leads:raw`, `leads:outreach`, Agent 2, Agent 6 | Частично готово, запуск ждёт `.env`, Redis, Telegram/Bitrix |
| Квалификация | Да | Agent 3: `cleaner`, `enricher`, `scorer`, `offer_gen` | Логика есть, реальный тест ждёт Redis и LLM/API |
| Персональное КП или лендинг | Частично | `offer_text`, `recommended_action`, `landing_page`, Agent 4, `site/` | Полноценный КП/лендинг-generator ещё не закрыт |
| CRM / карточка клиента | Да | Agent 5 `bitrix.create_lead()`, `notifier.notify()` | Код есть, запуск ждёт `BITRIX24_WEBHOOK_URL` и Telegram bot |
| Следующий шаг | Да | `QualifiedLead.recommended_action`, уведомление менеджеру, Bitrix comment | Есть на уровне поля и уведомления |
| Аналитика | Частично | source/UTM-поля в моделях, `analytics_reporter` CSV | Есть MVP-аналитика каналов, но нет связи с реальными сделками/выручкой |

Вывод: модель урока у нас уже заложена, но не полностью закрыта.

Главный разрыв:

```text
персональное КП или лендинг
```

Сейчас есть персональный первый оффер и следующий шаг, но нет полноценного модуля, который делает КП/лендинг по эталону и передаёт его в CRM как результат.

## 4. Где должен жить `Personal Offer Generator`

Нового агента создавать не нужно.

Правильное место внутри текущей системы:

| Функция | Где держать |
|---|---|

Важно: `agents/agent5_crm/proposal_trigger/__init__.py` сейчас пустой. Это не ошибка, а место для будущего маленького шага после MVP-доступов.

## 5. Маленький тест без платных сервисов и публикаций

Выбранный тест:

```text
один синтетический лид
-> ручная квалификация по структуре QualifiedLead
-> решение: нужен ли КП/лендинг
-> имитация CRM-карточки без Bitrix API
-> запись ожидаемого следующего шага
```

Почему этот тест безопасный:

- не нужен Anthropic/OpenRouter;
- не нужен Bitrix24 webhook;
- не нужен Redis;
- не нужны Telegram-сессии;
- не публикуются реальные посты;
- не используются реальные клиентские данные.

Что именно проверяет тест:

1. Может ли наша модель данных вместить весь путь лида.
2. Есть ли поле для персонального оффера.
3. Есть ли поле для лендинга.
4. Есть ли следующий шаг для менеджера.
5. Можно ли потом передать это в CRM и аналитику.

Минимальный тестовый пример:

```text
Источник: manual_fixture
Поток: B
Текст: Нужна рабочая документация и конструктив для реконструкции коммерческого помещения.
Ожидаемая квалификация: warm
Ожидаемый следующий шаг: запросить исходные данные и назначить созвон
КП/лендинг: нужен, но пока только как ручной черновик
CRM: создать тестовую карточку-заготовку без API
Аналитика: source=manual_fixture, landing_page=test_personal_offer
```

## 6. Где позже нужен `quality-comparator`

`quality-comparator` не создаём как нового агента.

Он нужен как будущий контрольный слой:

| Место | Что сравнивать |
|---|---|
| Agent 3 `offer_gen` | первый оффер против эталона: понятность, тон, следующий шаг |
| Agent 4 `creator` | КП/лендинг против эталонного человеческого КП |
| Agent 5 `proposal_trigger` | можно ли отправлять КП менеджеру/клиенту |
| Agent 5 `analytics_reporter` | отчёт по каналам против фактов расходов/сделок/выручки |
| Agent 6 `responder` | ответ в чатах против безопасного эталона коммуникации |

Простая схема качества:

```text
эталонный человеческий результат
-> черновик агента
-> список расхождений
-> правка промпта / шаблона / структуры
-> повторный тест
```

## 7. Что запускать не сейчас

Пока не запускать:

- реальные КП клиентам;
- реальные лендинги под конкретного клиента;
- Bitrix24 API;
- Telegram-отправки;
- платные LLM;
- массовый сбор лидов;
- новые агенты.

Причина: сначала нужно закрыть `.env`, Redis, Telegram-сессии и Bitrix24 для первого MVP.

## 8. Следующий маленький шаг

```text
заполнить обязательные поля .env
-> повторить SET/EMPTY/MISSING
```

```text

-> offer_text + recommended_action
-> CRM draft без Bitrix API
-> analytics row / отчет
```
