# Manual Universal Demand & Content Radar

Дата: 2026-05-12

Проект: `design-studio-lead-engine`

Статус: ручной MVP-срез. Массовый парсинг, платные API, скачивание чужого контента и публикации не запускались.

## 1. Правила среза

- Источники взяты из `content/library/sources/competitor-channel-watchlist.csv`.
- Анализировались только публичные страницы, видимые вручную.
- Чужие тексты, визуалы, сценарии, офферы и лид-магниты не копируются.
- Цель карточек: понять спрос, боли, рабочие форматы и безопасные действия для сайта, контента, лидогенерации и CRM-аналитики.
- Видимые метрики фиксируются только там, где они доступны публично и без автоматизированного сбора.

## 3. Что полезно для сайта

- Нужен блок `Что подготовить до консультации`: план/эскиз/описание объекта/город/тип помещения.
- Нужна отдельная логика для жилых и нежилых/коммерческих объектов, чтобы не смешивать разные боли в одну услугу.
- Нужна страница или блок `СРО и документы`, потому что рынок активно использует СРО как доверительный аргумент.
- Нужны FAQ и справочные страницы простым языком: кто готовит проект, кто согласовывает, что зависит от региона, что нельзя обещать заранее.
- Нужны кейсы с обезличенными фактами: тип объекта, город, задача, разделы, результат, без персональных данных.

## 4. Что полезно для контента

- Рубрика `Красные флаги в ТЗ`: для B2B-аудитории, ГИПов, проектировщиков и собственников бизнеса.
- Рубрика `Перепланировка без мифов`: короткие объяснения для собственников, дизайнеров и риелторов.
- Рубрика `Проектная организация простыми словами`: ПД, РД, КР, КЖ, КМ, СРО, экспертиза.
- Рубрика `Что спросить до ремонта/покупки/аренды помещения`: мост между контентом и заявкой.
- Разборы нормативных поводов допустимы только после проверки официальных источников.

## 5. Что полезно для лидогенерации

- Первый контакт должен просить не `оставьте телефон ради лид-магнита`, а минимальные исходные данные: тип объекта, город, задача, есть ли план/эскиз/документы.
- В CRM стоит разделять лиды минимум на: `apartment_replanning`, `commercial_nonresidential`, `building_project`, `reconstruction`, `b2b_partner`.
- Отдельный поток лидов стоит ждать от дизайнеров, риелторов, ремонтников и управляющих коммерческой недвижимостью.
- Для Agent 5 позже важно связать сигнал с источником: `source_id`, `segment`, `topic`, `first_touch_channel`, `recommended_next_step`.

## 6. Нужно ли дорабатывать Agent 1 Scout

Да, но маленько и без создания нового агента.

Нужна не новая логика сбора, а единый выход:

```text
Agent 1 Scout -> SignalCard -> Agent 4 / Agent 2 / Agent 5 / Site
```

Минимальная доработка после ручного MVP:

- добавить `SignalCard` как обязательный формат research-выхода Agent 1;
- добавить лимит: 5-10 карточек за ручной цикл;
- добавить поле `handoff_to`, чтобы карточки не лежали мёртвым списком;
- добавить статус `new/reviewed/approved/used/rejected`.

## 7. Нужен ли отдельный Radar Agent

Пока нет.

На текущем этапе достаточно протокола `Universal Demand & Content Radar` внутри Agent 1 Scout.

Отдельный Radar Agent стоит рассматривать позже, если:

- ручной процесс повторится минимум 2-3 раза;
- карточки начнут регулярно давать идеи для сайта, контента и лидов;
- появятся понятные метрики полезности;
- будет ясно, какие источники можно автоматизировать безопасно и без лишнего бюджета.

## 9. Что потом делать с этой информацией

Чтобы SignalCard не превратились в склад идей, у каждой карточки должен быть следующий путь:

```text
SignalCard -> решение -> задача -> проверка -> статус used/rejected
```

Практический порядок:

1. Раз в неделю выбрать 3-5 карточек со статусом `reviewed`.
2. Разложить их по `handoff_to`: сайт, Agent 4, Agent 2, Agent 5 или Agent 6.
3. Для каждой карточки создать одно маленькое действие, а не большой проект.
4. После действия поменять статус карточки на `used` или `rejected`.
5. Если карточка дала заявку, тему, блок сайта или полезный вывод, Agent 5 позже связывает это с каналом и результатом.

Пример:

```text
карточка про документы до консультации
-> handoff_to: Site
-> задача: добавить блок "Что подготовить до консультации"
-> проверка: блок есть на сайте, без обещаний и без персональных данных
-> status: used
```

Правило для MVP: одна карточка должна превращаться в одно маленькое действие. Если действие слишком большое, карточку нужно разбить.
