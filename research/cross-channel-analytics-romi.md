# Сквозная аналитика каналов

Дата: 2026-04-30
Проект: Lead Intelligence Pipeline, ООО «Вектор Плюс-Про»
Статус: архитектурное решение перед расширением каналов

## Главная мысль

Система не должна быть просто парсером лидов.

Правильная цель:

```text
источник -> лид -> заявка -> сделка -> выручка -> ROMI
```

Простыми словами: нам важно знать не только “откуда пришёл лид”, но и какой источник реально принёс деньги.

Если канал даёт много лидов, но нет сделок, это плохой канал.
Если канал даёт мало лидов, но они дорогие и закрываются в сделки, это может быть хороший канал.

По скриншотам лекции видно, что пример системы не останавливается на лидах.

## Чего сейчас не хватает

Главный пробел: система пока больше про лиды, чем про окупаемость каналов.

Не хватает:

| Чего не хватает | Зачем нужно |
|---|---|
| Единый справочник каналов | Чтобы не писать “tg”, “telegram”, “тг” как разные источники |
| UTM-метки | Чтобы понимать, какая реклама или пост привели лид |
| Расходы по каналам | Без расходов нельзя считать стоимость лида и ROMI |
| Связь лида со сделкой Bitrix24 | Чтобы видеть путь от заявки до денег |
| Поле выручки | Чтобы считать, сколько канал принёс денег |
| Поле маржи | Чтобы считать не только выручку, но и реальную прибыль |
| Отчёт по каналам | Чтобы видеть, что масштабировать, а что выключать |
| Метка первого касания | Чтобы не потерять реальный первоисточник |
| Метка последнего касания | Чтобы видеть канал, который довёл до заявки |

## Минимальная схема данных

Каждый лид должен хранить источник и путь до сделки.

Минимальная цепочка:

```text
raw_lead
-> qualified_lead
-> Bitrix24 lead
-> Bitrix24 deal
-> revenue
-> ROMI report
```

## Как считать показатели

Минимальные формулы:

```text
visit_count = количество визитов из канала
lead_count = количество лидов из канала
qualified_count = количество целевых лидов из канала
meeting_count = количество встреч из канала
deal_count = количество сделок из канала
revenue = сумма выручки по сделкам канала
profit = revenue - ad_cost
ad_cost = расходы на канал
cost_per_lead = ad_cost / lead_count
cost_per_qualified_lead = ad_cost / qualified_count
cost_per_meeting = ad_cost / meeting_count
cost_per_deal = ad_cost / deal_count
payback_ratio = revenue / ad_cost
ROMI = (revenue - ad_cost) / ad_cost * 100
profitability = (revenue - ad_cost) / revenue * 100
CR_visit_to_lead = lead_count / visit_count * 100
CR_lead_to_qualified = qualified_count / lead_count * 100
CR_qualified_to_meeting = meeting_count / qualified_count * 100
CR_meeting_to_deal = deal_count / meeting_count * 100
CR_visit_to_deal = deal_count / visit_count * 100
```

Если расходов нет, например SEO или рефералы, канал всё равно нужно учитывать, но ROMI считать осторожно.
Для таких каналов сначала считаем не рекламный расход, а трудозатраты или комиссию.

Простое различие:

```text
окупаемость = во сколько раз канал вернул вложенные деньги
рентабельность = какая доля выручки осталась прибылью после расходов канала
```

Пример:

```text
расход = 10 000
выручка = 50 000
прибыль = 40 000
окупаемость = 50 000 / 10 000 = 5x
ROMI = 400%
рентабельность = 40 000 / 50 000 * 100 = 80%
```

Решение по каналу должно смотреть не только на число лидов, а на деньги:

```text
scale = канал даёт сделки, положительный ROMI и хорошую рентабельность
keep = канал окупается, но не выглядит как явный лидер
watch_quality = лиды есть, сделок нет
check_tracking_or_pause = деньги тратятся, лидов нет
pause_or_fix = канал убыточный
no_data_yet = данных пока нет
planned = канал ещё не запущен
```

## Важное правило

Нельзя запускать платную рекламу без учёта источника.

Минимум перед запуском любой рекламы:

```text
лендинг или форма
UTM-метки
источник в Bitrix24
этапы воронки: лид, квалификация, встреча, сделка
таблица расходов
ответственный менеджер
SLA ответа
```

Иначе мы получим заявки, но не поймём, окупился ли канал.

## Итог

Идея из лекции правильная для нашего проекта.

Но добавлять её нужно как слой сквозной аналитики, а не как хаотичный список новых источников.

Наша финальная логика:

```text
источник -> первое касание -> лид -> квалификация -> Bitrix24 -> сделка -> выручка -> ROMI -> решение по каналу
```
