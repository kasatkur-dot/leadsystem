# AGENTS.md — правила Codex для design-studio-lead-engine

## Базовые правила

- Никогда не удалять папки `/System`, `/Applications`, `/Users`, `/Library`, `/Users/yanika/Documents`.
- Никогда не показывать значения секретов из `.env`; для проверок использовать только `SET` / `EMPTY`.
- Перед любой задачей сначала читать `REPORT.md`.
- Если задача связана с ремонтом агента, сначала читать `REPORT.md` и `CLAUDE.md`.
- После рабочих изменений обновлять `REPORT.md`: что сделано, что проверено, что осталось, следующий маленький шаг.
- Если деталей много, не раздувать `REPORT.md`: короткий итог писать туда, а подробности переносить в `tasks/session-notes.md`.
- Если появляется идея `применим позже`, обязательно записывать её в `content/pipeline/future-opportunities-backlog.md`: источник, польза, когда применять, почему не сейчас, риск и статус.
- Не переписывать всю систему, если сломалась одна часть.
- Чинить минимальную сломанную часть после проверки логов, `.env`, Redis и очереди нужного агента.
- Вести пользователя строго по шагам, простыми словами, без перескакивания через непроверенные этапы.
- Перед задачами с секретами, внешними сервисами, CRM, публикациями, deploy, MCP/API, LLM-вызовами, платежами, массовым сбором или реальными пользовательскими данными читать `agents/Агент безопастности/AGENT_SECURITY.md`.

- Память проекта и журнал: `REPORT.md`.

- Старый `sales_ai` закрыт как активный проект и перенесён в архив `../_archive_sales_ai_2026-05-final`.
- Agent 4 теперь внутренний контент-движок в `agents/agent4_publisher/`.
- `agents/Агент безопастности/AGENT_SECURITY.md` — контрольный слой безопасности для всех 6 агентов, checker/dashboard и интеграций. Это не Agent 7.

## Правила по новому сайту

- Перед задачами по новому сайту сверяться с `docs/new-site-structure.md`, `docs/new-site-design-brief.md` и `docs/geo-ai-source-of-truth.md`.
- При доработке сайта не забывать медиа-слой: промо-блок/анимация, визуализация пути клиента, social preview, будущие короткие ролики для MAX/Telegram/VK/Дзена.
- На сайте не использовать автозвук; звук допустим только в отдельном промо-видео, которое пользователь запускает сам.
- HyperFrames, Remotion, HeyGen или аналоги использовать только позже и только после отдельного подтверждения, без платных запусков по умолчанию.
- Сайт должен выглядеть как публичный лендинг, а не как внутренний документ: не выводить в клиентские блоки фразы `первая версия`, `плейсхолдер`, `social preview`, `потом усилим`.
- В быстрых контактах использовать телефон, email, MAX-аккаунт для сообщения, MAX-канал `СИЛА Проекта`, Telegram, Яндекс Карты и 2GIS; `СРО / протокол` не держать как быстрый контакт.
- `СИЛА Проекта` всегда подписывать как MAX-канал.
- Реальные проектные скрины вставлять только после обезличивания адресов, ФИО, телефонов, сумм, штампов, кадастровых номеров и служебных отметок.

## Запреты

- Не использовать курс, Jarvis, прогрев, лекции или ученические материалы из архивного `sales_ai`.
- Не создавать формы захвата, FSM-квизы, обязательный email/телефон или лид-магнит в обмен на контакт.
- Не создавать седьмого агента.
- Не дублировать CRM, Bitrix24 или аналитику вне Agent 5.
- Не запускать всю систему, пока не проверены `.env`, Redis, Telegram-сессии и Bitrix24.
- Не обходить `AGENT_SECURITY.md`, если задача может затронуть деньги, продакшен, секреты, внешнюю коммуникацию, MCP/API, LLM или данные клиентов.


## Секреты и локальные ключи

Применять общий протокол `../PROJECT_SECRET_RULES.md`.

Если Яника передает токен/API-ключ для локальной настройки, не повторять его в ответе, не писать в `REPORT.md`, `README.md`, `AGENTS.md`, `CLAUDE.md`, `tasks/session-notes.md`, логи или публичные файлы. Хранить только локально в `.env`, `.env.local`, gitignored `secrets/` или системном хранилище.

Для проверок `.env` показывать только `SET` / `EMPTY`. Перед GitHub, деплоем, архивом или передачей другому агенту проверить `.env`, Redis/Telegram session-файлы, Bitrix24 webhooks, credentials, `logs/` и `*.jsonl`.

## Агент безопасности

Файл управления безопасностью: `agents/Агент безопастности/AGENT_SECURITY.md`.

Применять как контрольный слой:

- к Agent 1/2 при разведке и сборе источников;
- к Agent 3 при LLM-квалификации и обработке входящих данных;
- к Agent 4 при публикациях, текстах, медиа и внешних каналах;
- к Agent 5 при Bitrix24, Telegram, аналитике, сделках и CRM-данных;
- к Agent 6 при outreach и первом касании;
- к checker/dashboard при отображении статусов без секретов и без внешних вызовов.
